# Mapping Engine

## Context and Design Philosophy

This component is the one place MIDI meaning becomes Dragonframe meaning. Each mapping entry translates a normalized MIDI event into zero-or-one Dragonframe OSC message, through a single interface (`event -> Optional[OscMessage]`) that keeps this component swappable without touching MIDI I/O, OSC I/O, or the UI.

A second, separate interface, `process_keystroke(event) -> Optional[KeyCombo]`, exists alongside `process()` for the small number of entries whose Dragonframe function has no OSC equivalent at all (`docs/llds/keystroke-output.md`). It is not a replacement for `process()` and does not change `process()`'s return type or existing callers — the two methods are called independently, for the same event, by whatever drives the event loop (`app.py`). In this phase, the jog wheel is the only entry that produces anything from `process_keystroke()`; every other event returns `None` from it. This is a hard-coded default-map behavior, not a user-configurable target choice, so it doesn't conflict with the HLD's "one target per mapping entry" decision, which governs the general editable target-type model (Phase 2), not this phase's opinionated jog wheel default.

A third interface, `process_websocket(event, now, axis_positions) -> Optional[WebSocketCommand]`, exists for the small number of entries whose Dragonframe function is reachable only through the WebSocket output path (`docs/llds/websocket-output.md`) — Stop, Cycle, Solo 1–8, and Marker ◄/►. Unlike the jog wheel's keystroke output, these controls' WebSocket target **replaces** their prior OSC target rather than firing alongside it — each entry has exactly one target, still matching the HLD's "one target per mapping entry" principle; it's simply a WebSocket target instead of an OSC one for these four. See WebSocket-Targeted Controls below.

Unlike the original static-table design, entries are now editable (add/edit/remove, enable/disable, MIDI-learn) and persisted through a Preset Store, per `docs/high-level-design.md`. The table ships pre-loaded with the nanoKONTROL Studio's opinionated default map (mirroring the prior prototype's default map), so a user who never opens the mapping view sees identical behavior to a fixed table.

**Phase scope** (per `docs/high-level-design.md § Delivery Phasing`): this LLD describes the target design. **App Delivery Phase 1** implements the "OSC axis (direct)" target type, as the **default** for the 8 faders, plus **bank derivation** — knobs and the Mute button automatically follow their bank's fader assignment rather than being independently configurable (see Bank Derivation below); Solo is WebSocket-targeted and independent of bank state (see WebSocket-Targeted Controls below). Transport/marker/track buttons and Record/Select are unaffected and keep their existing targets. The jog wheel drives frame-by-frame timeline stepping (see Jog Wheel Frame Stepping below); Return to Zero remains unmapped (see the Opinionated Default Map below). The general editor (MIDI-learn, add/remove, presets, independently retargeting a knob/button) is **App Delivery Phase 2**.

## Target Types

Each entry has exactly one target, chosen by the user (or left at its default):

- **OSC action** — one of Dragonframe's fixed named commands (`shoot`, `play`, `live`, `mute`, `black`, `delete`, `shootVideoAssist`, and the further commands listed in `docs/dragonframe-messages-research.md` not yet defaulted to any control).
- **OSC encoder channel** — `/dragonframe/encoder/{n}` / `/dragonframe/encoderReset/{n}`, requiring the user to separately wire that channel to an axis inside Dragonframe's Arc workspace. The fallback for a fader/knob/Mute whose bank has no axis assigned; no longer the default for faders. Solo has no OSC encoder fallback — it is WebSocket-only, unconditionally (see WebSocket-Targeted Controls below).
- **OSC axis (direct)** — addresses a discovered axis by name directly (see below). **The default target type for faders in App Delivery Phase 1.** Knobs and Mute reach this same mechanism only through bank derivation, not as an independently selectable target.
- **Custom OSC path** — arbitrary address/argument, restoring the old prototype's escape hatch. Not implemented before App Delivery Phase 2.
- **WebSocket** — one of Dragonframe's WebSocket-recognized commands (`docs/dragonframe-websocket-research.md`), sent via `docs/llds/websocket-output.md`'s adapter. Hard-coded to Stop, Cycle, Solo 1–8, and Marker ◄/► in this phase, not independently user-selectable — see WebSocket-Targeted Controls below.

## Fader Axis Mode (Default) vs. OSC Encoder Mode

- Every fader starts in **axis mode**, with no axis name selected. In this state, moving the fader produces **no OSC output at all** — there is no fallback to the opinionated encoder target while unconfigured. An axis name is inherently project-specific; unlike an encoder channel number, there is no meaningful placeholder to fall back to until the user configures one.
- Switching a fader's target type to **OSC encoder** enters encoder mode: the fader (and its bank, see below) behaves exactly as documented in the Opinionated Default Map table's encoder-fallback column.
- Switching back to **OSC axis** re-enters axis mode; if no name has been picked yet (or was cleared), it again sends nothing until one is chosen.
- This is tracked by `MappingEngine` itself, not only the Mapping View UI — it changes actual dispatch behavior (whether an event produces OSC output at all), not merely what the table displays. `set_axis_target` implies axis mode; `clear_axis_target` implies encoder mode, in addition to their existing effects on the stored target and dedup state.

## Bank Derivation

The 8 channel strips are grouped into **banks**. Bank N = Fader N, Knob N, Mute N. Solo N is grouped with the same 8 channel strips physically, but is **not** bank-derived — it always sends its WebSocket target regardless of Fader N's axis state (see WebSocket-Targeted Controls below); it is called out separately here rather than folded into "Bank N" to avoid implying its behavior depends on Fader N, the way Knob N's and Mute N's do. Record N and Select N are not part of a bank — no per-axis OSC action exists for them (`docs/dragonframe-messages-research.md`), so they remain unmapped regardless of bank state, unchanged from today.

- **Only Fader N's target is directly configurable** — the OSC axis (direct) picker described above and in `docs/llds/app-ui.md`. Knob N and Mute N have no independent target selection; their effective target is derived from Fader N's current state, recomputed on every event they produce, not fixed at the moment Fader N's axis was chosen.
- **Fader N has a real axis name assigned:**
  - Knob N sends `/dragonframe/axis/{axisname}/stepPosition,f (delta)` on every distinct value once a prior reading exists to compare against, where `delta = (raw_value - previous_raw_value) * 0.1` — the change since Knob N's own last reported value, scaled down by a fixed factor of `0.1` axis-position units per MIDI raw-value increment, for fine adjustment rather than a whole-unit-per-tick jump. The first reading after Fader N's axis is assigned (or after any encoder↔axis mode transition, see below) establishes this baseline only and produces no send, since there is nothing yet to compute a change against. No debounce; the scale factor is a fixed constant, not user-configurable. **Delta must be computed against the knob's own previous reading, not its distance from a fixed center.** Dragonframe accumulates each `stepPosition` onto the axis's current position; a knob's physical rotation typically produces several intermediate MIDI messages, so a formula based on absolute distance from a fixed reference would resend a large, nearly-constant value on each one — producing runaway movement rather than a proportional nudge.
  - Mute N sends `/dragonframe/axis/{axisname}/setZero` on press.
  - **`setZero` is a calibration command, not a movement command.** It redefines the axis's zero reference point to its current position; it does not move the axis to a previously-stored zero. Dragonframe's OSC surface has no command that does the latter (`docs/dragonframe-messages-research.md`).
- **Fader N has no axis assigned** (default state, or explicitly switched to OSC encoder): Knob N and Mute N fall back to their table-listed static defaults (`/dragonframe/encoder/{9-16}`, `/dragonframe/encoderReset/{1-8}`) — unchanged from today's behavior.
- **A repeated identical raw value produces a computed delta of `0` and is not resent** — this falls directly out of the delta formula (a repeat means `raw_value == previous_raw_value`, so the raw difference — and its scaled result — is `0`), with no separate dedup check needed.
- **Knob N's stored "previous reading" spans two different value semantics** — a normalized absolute float (0.0–1.0) in encoder mode, versus a raw MIDI int in derived-axis mode. Comparing a stale value from one semantic against a fresh value from the other, right at a mode transition, would produce a meaningless delta. Assigning Fader N's first axis name (entering axis mode from encoder mode) or clearing it (leaving axis mode for encoder mode) therefore also discards Knob N's stored reading — mirroring the same "switching a target discards prior dedup state" precedent already established for the fader itself, and forcing the next knob event to re-establish a fresh baseline rather than diff against a stale, incompatible value. Picking a *different* axis name while Fader N is already in axis mode does not discard it, since the delta computation doesn't depend on which axis is targeted.

### Knob N's position clamping (`MAP-BANK-008`, `MAP-BANK-009`)

Since the fader defines the axis's usable range (its configured `min`/`max`), Knob N's cumulative nudges must not be able to push the axis outside that same range — otherwise "fine-tune below the fader's floor" would silently produce an out-of-range `gotoPosition`-equivalent via repeated `stepPosition` calls, with no floor at all.

- The engine tracks, per fader key, a running position estimate built from its own sends: it starts at the fader's own last `gotoPosition` value (or, if the knob is nudged before the fader has ever sent one, the fader's configured lower bound), and updates on every knob nudge that is not itself clamped away.
- Before applying a nudge, the engine consults `AxisDiscovery.axes` (threaded through as `MappingEngine.process()`'s `axis_positions` parameter) for a live reading of that axis's actual reported position. **A live reading, when present, is authoritative** and is used in place of the internal estimate — Dragonframe's own broadcast reflects reality even if some other client (or an unexpected clamp/rounding on Dragonframe's side) moved the axis independently of DragonMIDI's estimate. The internal estimate is a fallback only, used when no live reading exists yet for that axis name (e.g. "Output Motor Positions" isn't enabled in Dragonframe, or no broadcast has arrived since startup/reconnect).
- The effective bounds are `low, high = sorted((fader.min_value, fader.max_value))`, order-independent since `MAP-AXIS-002` allows `min > max`.
- If `current_position + delta` falls within `[low, high]`, the full delta is sent unchanged, and the tracked estimate advances to the new position.
- If it would fall outside, the sent delta is reduced to exactly what's needed to land on the boundary (`clamped_position = max(low, min(high, current_position + delta))`, `clamped_delta = clamped_position - current_position`); if `current_position` is already at that boundary, the reduced delta is `0` and nothing is sent at all (distinct from `MAP-BANK-005`'s "repeated raw value" no-send case, though the observable effect — no OSC message — is the same).
- **Floating-point care**: the reduced-delta path recomputes `clamped_delta` via subtraction of two position floats, which can reintroduce floating-point error even in cases that turn out not to need clamping. The common (non-clamped) path therefore uses the original, exactly-scaled `delta` value directly rather than routing it through the same subtraction, avoiding spurious precision loss on ordinary nudges.

## OSC Axis (Direct) Target (Fader — `gotoPosition`)

- The user picks an axis name from the list the OSC Listener has discovered via `getAllPosition` (see `docs/llds/osc-io.md`), plus a **min** and **max** position value.
- For a continuous-absolute MIDI source (a fader), the engine sends `/dragonframe/axis/{axisname}/gotoPosition,f (position)` on every distinct MIDI value, with no debounce — position is computed as `min + normalized_value * (max - min)`, matching the same "continuous, no debounce" handling already used for OSC encoder targets.
- Min/max are **entered by the user, not discovered** — Dragonframe has `setLimits` but no `getLimits` over OSC, so an axis's practical range cannot be read back (`docs/dragonframe-messages-research.md`).
- **Setup precondition, not a DragonMIDI-enforced invariant:** the target axis's Function must be `Manual` (or otherwise not require a real connected device) for `gotoPosition` to actually move it. Empirically confirmed: an axis with `Function: Normal` and a real hardware `Connect` type (e.g. ArcMoco) but no physical device attached accepts `gotoPosition` without error but never actually moves, even with Dragonframe's "Ready to Capture" mode enabled; switching to `Function: Manual` fixed this immediately (`docs/dragonframe-messages-research.md § Axis Discovery and Direct Addressing`). DragonMIDI has no way to detect an axis's Function type over OSC, so this is documented user-facing guidance, not something the mapping view can validate or warn about.
- **Axis name selection is picker-only, not free text, and is gated only at selection time.** The user selects from the OSC Listener's currently-discovered axis list at the moment they configure the mapping; there is no way to type an arbitrary/undiscovered name into a direct-axis mapping, and there is no ongoing re-validation afterward (see the stale-reference bullet below — the restriction is a one-time UI gate, not a continuously-enforced invariant). A free-text field would let a typo produce a mapping that silently sends `gotoPosition` to a name Dragonframe doesn't recognize, with no way for DragonMIDI to detect the mistake.
- **Rescan to pick up newly added axes.** If the user adds an axis in Dragonframe after DragonMIDI has already discovered its axis list (or after switching to a different project), the on-demand "Rescan"/refresh action (`OSC-DISCOVER-003`, `docs/llds/osc-io.md`) re-queries `getAllPosition` and updates the discoverable list without requiring a DragonMIDI restart.
- **No validation on min/max ordering.** `min + normalized_value * (max - min)` is a well-defined linear interpolation for any real `min`/`max` pair, including `min > max` (a legitimate reversed/inverted mapping) or `min == max` (a constant output regardless of fader position, which can be a deliberate way to always send one fixed value). Neither is rejected.
- **Stale axis references are an accepted, undetected gap.** If the mapped axis name stops being reported by Dragonframe (the axis was deleted, or a different project with different axes was loaded), the engine keeps sending `gotoPosition` to that name regardless — Dragonframe presumably ignores it silently. DragonMIDI does not compare a mapping's target name against the current discovered list to warn about this, matching the project's general pattern of accepting rare, self-evident gaps rather than building detection machinery for them. Rescanning (above) helps a user find new axes; it does not retroactively validate mappings made against axes that have since disappeared.
- **Switching a mapping entry's target type discards the previous target's configuration.** Retargeting a fader from, say, an OSC encoder channel to an OSC axis (direct) replaces the entry's target-specific fields entirely — the old encoder channel number is not preserved for a later switch-back. This matches "one target per entry, not multiple" cleanly and avoids stale, hidden configuration silently persisting in a mapping entry (and potentially a saved preset). Implemented in the Mapping View UI (`docs/llds/app-ui.md`'s `UI-MAP-003`).
- **Reverting from OSC axis back to OSC encoder is a full clear, not a swap.** `MappingEngine.clear_axis_target(key)` removes the fader's axis-target entry (and its dedup state) entirely; `process()` then falls back to the opinionated `OPINIONATED_MAP` lookup for that key on the very next event, same as if the axis target had never been set. There is no hidden "remembered" encoder-channel override to restore — a fader always reverts to its one opinionated encoder target, symmetric with the discard-on-switch behavior above.

## Jog Wheel Frame Stepping

The jog wheel (CC 110) reports movement using the nanoKONTROL Studio's KORG sign-magnitude relative encoding, reused byte-for-byte from the prior prototype's decode (`relative_sign_magnitude` trigger): a message's raw value `1–63` means clockwise rotation by that magnitude, `65–127` means counterclockwise rotation by `raw_value - 64`, and `0` or `64` both mean no movement.

- **Direction only, magnitude ignored.** Every clockwise message (`1–63`) sends `/dragonframe/stepForward` exactly once; every counterclockwise message (`65–127`) sends `/dragonframe/stepBackward` exactly once. The message's magnitude is not used to send multiple steps or to scale anything — one physical detent is one frame, regardless of how fast the wheel was spun. A message of `0` or `64` produces no OSC output.
- **No debounce, no dedup.** Unlike button-type entries (`MAP-DEBOUNCE-001`'s 80ms window) or continuous-absolute entries (skip-on-repeated-value), every qualifying jog wheel message fires immediately and independently — each message already represents one discrete physical detent, not a bounce or a resend of an unchanged value. A rapid run of identical raw values (e.g. several `1`s in a row from a fast spin) is several distinct detents, not one repeated reading, and each fires its own step.
- **Stateless.** The jog wheel entry holds no per-control state (no `_previous_value`, `_pressed_state`, or `_last_fired` entry) — its OSC output is a pure function of each message's raw value. There is nothing to clear on MIDI reconnect for this control.
- **Channel match still applies.** As a CC-sourced control, the jog wheel is still subject to `MAP-TABLE-001`'s channel-16 match invariant like every other CC entry — a jog wheel message on any other channel does not match.
- **Implementation shape:** the jog wheel's target address is conditional on the message's sign, which doesn't fit `_MapEntry`'s fixed-address shape used by the rest of `OPINIONATED_MAP`. It is special-cased in `MappingEngine.process()` ahead of the generic table lookup, the same way the Scene button and fader-axis-mode paths already are, rather than adding a second address field to `_MapEntry` for one control.

### Keystroke Output (Arc Motion Control)

The main timeline's `stepForward`/`stepBackward` OSC commands have no effect while the Arc Motion Control workspace is focused (`docs/high-level-design.md § Problem`). Dragonframe's only reachable equivalent there is a distinct action, "Step Moco Forward"/"Step Moco Back," with no OSC message of its own, only a Hot Key binding (`Option+Shift+Right`/`Option+Shift+Left` by default).

- On the same event and using the same sign-magnitude direction decode as the OSC output above, `MappingEngine.process_keystroke()` returns `KeyCombo(frozenset({"alt", "shift"}), "right")` for a clockwise message and `KeyCombo(frozenset({"alt", "shift"}), "left")` for a counterclockwise one; `0`/`64` and any non-jog-wheel event return `None`. `"alt"` is the cross-platform name for the physical key Dragonframe's default binding calls "Option" on macOS.
- **Both outputs fire for the same message.** A single jog wheel MIDI message produces one `process()` call and one `process_keystroke()` call from the event loop, each independently evaluated — there is no attempt to detect which of the two would actually have an effect (main timeline vs. Arc workspace focused) and suppress the other. The OSC command is a no-op when Arc has focus; the synthesized keystroke is presumably a no-op (or at least harmless) when the main timeline has focus and Arc isn't open. This mirrors `docs/high-level-design.md § Non-Goals`' decision not to build frontmost/focus-detection machinery.
- **Assumes the default Hot Key binding.** If the user has remapped "Step Moco Forward"/"Step Moco Back" away from `Option+Shift+Right`/`Left` in Dragonframe's own Hot Keys preferences, the synthesized keystroke will silently do nothing (or trigger whatever else that combo is now bound to) — DragonMIDI has no way to read Dragonframe's actual configured bindings, the same "documented precondition, not detected" treatment as the axis `Function: Manual` requirement.
- **No debounce or dedup**, matching the OSC output's `MAP-JOG-004` and `docs/llds/keystroke-output.md`'s "no debounce or dedup at this layer" — every qualifying message produces its own keystroke, independent of prior messages.
- **Stateless, like the OSC-side jog wheel handling.** `process_keystroke()` reads only the event's raw value; it neither consults nor mutates `_previous_value`/`_pressed_state`/`_last_fired`, so it needs no reconnect-clear handling and `tracked_controls()` is unaffected by it (`MAP-JOGKEY-007`).

## WebSocket-Targeted Controls

Four controls target `docs/llds/websocket-output.md`'s adapter instead of OSC, replacing their prior OSC default entirely (not firing alongside it, unlike the jog wheel's dual OSC+keystroke output above). All four are press-edge controls and reuse the same edge-detection and 80ms debounce logic as any other button entry (`MAP-DEBOUNCE-001`) — `process_websocket()` shares `_pressed_state`/`_last_fired` state with `process()`, keyed by the same `_Key` tuples. This relies on `MAP-WS-009` (below): these four keys are removed from `OPINIONATED_MAP` entirely, so `process()` never matches them and there is no collision in reusing the same dicts across the two methods for the same key.

**Terminology note:** `process_websocket()` *returns* a `WebSocketCommand`; whether it's actually delivered to Dragonframe is `docs/llds/websocket-output.md`'s concern, not this component's. If no WebSocket connection is currently active, `WS-SEND-004` silently drops it — from this LLD's point of view, the press still "produces a command" every time; "produces" below should be read that way, not as a delivery guarantee. This is consistent with `docs/high-level-design.md`'s decision that WebSocket output fails silently with no dedicated indicator — a press with no active connection is simply not observable to the user as having done anything, the same as any other silent-fail case in this app.

- **Stop (CC 42) → `E-Stop`.** Produces the bare trigger `WebSocketCommand("E-Stop")` on press. Removed from `OPINIONATED_MAP`; `process()` no longer matches this key at all (falls through to `MAP-TABLE-005`'s no-match path).
- **Solo N (CC 32–39) → `select-AXN`.** Produces `WebSocketCommand(f"select-AX{N}")` on press, where `N` is the button's own fixed number (1–8) — unconditional, independent of Bank N's fader/axis state (see Bank Derivation above) and independent of the discovered axis count (unlike Cycle, below). If the project has fewer than `N` axes, Solo N addresses an axis identity Dragonframe doesn't have; this is accepted the same way a stale OSC axis-name reference is accepted elsewhere in this LLD — DragonMIDI doesn't validate against a live count before sending. Removed from bank derivation entirely; `_process_bank_derived` no longer has a Solo branch.
- **Cycle (CC 46) → `select-AXn`, cycling.** Produces `WebSocketCommand(f"select-AX{n}")` on press, where `n` advances by one (1-indexed) on every press and wraps modulo the current axis count. Removed from `OPINIONATED_MAP`.
  - The axis count is `len(axis_positions or {})`, reusing the same `axis_positions` parameter `process()` already receives from `AxisDiscovery.axes` (`docs/llds/osc-io.md`) — **not** a count read from the WebSocket connection's own `replaceInputList`, which `docs/llds/websocket-output.md`'s `WS-CONN-006` deliberately never parses. This rests on an **unverified assumption**: that Dragonframe's WebSocket-side `AX1`/`AX2`/… numbering corresponds, in order, to the axis names OSC's `getAllPosition` discovers. Both were confirmed to report the same two axes in this project's testing, but the correspondence itself — as opposed to each mechanism's individual behavior — was never directly tested.
  - `len(axis_positions or {})` collapses `AxisDiscovery`'s two distinct empty states (`None` = never queried; `{}` = queried, zero axes — `docs/llds/osc-io.md`) into the same `0` count. During the ~2-second discovery-timeout window right after a fresh MIDI/OSC reconnect, a Cycle press therefore behaves identically to "genuinely zero axes": no command produced. This is an accepted simplification, not a distinction worth threading through — either way, there's nothing yet to cycle to.
  - If the axis count is `0`, Cycle presses still register as a normal press edge against `_pressed_state`/`_last_fired` (the debounce window still applies to the next press) but produce no WebSocket command and do not advance `_cycle_index` — there is nothing to cycle through.
  - `MappingEngine` tracks one new piece of state, `_cycle_index: int`, starting at `-1` (nothing selected yet). Each press computes `_cycle_index = (_cycle_index + 1) % axis_count` and sends `select-AX{_cycle_index + 1}`. Cleared on MIDI reconnect, matching the rest of the engine's state (see State below).
  - **This reconnect-clear is MIDI-scoped, not WebSocket-scoped, and the two can drift.** `_cycle_index` resets whenever the MIDI Input Adapter reconnects; Dragonframe's own server-side "currently selected axis" is tied to the WebSocket connection, which can reconnect independently (e.g. Dragonframe regains OS focus with no MIDI hiccup at all — `docs/dragonframe-websocket-research.md`'s focus-triggered reconnect behavior). After either reconnect happens without the other, Cycle's next press may not match whatever axis Dragonframe still considers selected from before. **Accepted as a known limitation**, not built around — matches this project's established pattern of accepting rare, self-evident state-divergence gaps (e.g. stale OSC axis references, above) rather than adding cross-component connection-change notification machinery that neither `docs/llds/websocket-output.md` nor this LLD currently has.
  - Cycle's index is entirely independent of Solo's presses in the sense that pressing Solo N does not read or write `_cycle_index`, and vice versa — **but both write the same Dragonframe-side "currently selected axis"**, so they are not independent in effect: Solo N pressed after Cycle (or vice versa) changes what a subsequent Marker press (below) will act on. "Independent" here means "no shared DragonMIDI-side state," not "no shared effect on Dragonframe."
- **Marker ◄/► (CC 61 / CC 62) → `Jog All`.** Previous Marker produces `WebSocketCommand("Jog All", operation="+", params=(-1,))`; Next Marker produces `WebSocketCommand("Jog All", operation="+", params=(1,))`, one increment per press (matching the jog wheel's "one detent, one step" precedent, not a continuous stream). Both removed from `OPINIONATED_MAP`.
  - `Jog All` is a generic, not-axis-specific command — it acts on whichever axis Dragonframe's own server-side state currently considers "selected," last set by a `select-AXn` send from **either** Solo or Cycle (both feed the same Dragonframe-side state — see above). **Marker ◄/► therefore does nothing until Solo or Cycle has selected an axis at least once** in the current Dragonframe session; this is an accepted dependency between these controls, not a bug. DragonMIDI has no way to observe whether that prior selection actually landed (`WS-CONN-006` parses nothing back), so this dependency is inherently untestable from DragonMIDI's own state — verifiable only by observing Dragonframe itself.
  - Chosen over the originally-proposed `jog-AXn` (`docs/nanokontrol-mapping-proposal.md`): `Jog All` has direct log-verified confirmation (`docs/dragonframe-websocket-research.md`), while `jog-AXn` was never tested, only inferred by naming-pattern analogy with `select-AXn`/`Step`. Using `Jog All` also means `MappingEngine` doesn't need to track "which axis is currently selected" itself — Dragonframe already does, and `select-AXn`'s confirmed retargeting behavior (`docs/dragonframe-websocket-research.md`) means DragonMIDI doesn't need to duplicate that state.

**No cross-output ordering guarantee.** `process()`'s OSC send and `process_keystroke()`'s keystroke send are synchronous on the calling thread; `process_websocket()`'s result is handed to the WebSocket adapter's `send()`, which itself hands off to a separate event-loop thread (`docs/llds/websocket-output.md § Runtime Model`, `WS-RUNTIME-001`). Nothing in this app guarantees that a WebSocket command reaches Dragonframe before or after an OSC/keystroke send for a later MIDI event — two rapid presses (e.g. Cycle then Marker) could theoretically be observed out of order at Dragonframe. Accepted: MIDI events are still drained and dispatched to all three `process*()` calls in strict arrival order on the Qt thread, and the async handoff is expected to be fast in practice; building a cross-output ordering guarantee would mean serializing all three output paths (OSC, Keystroke, WebSocket) behind one send queue, a real architectural change out of proportion to a theoretical race.

**Same axis-count snapshot within one event.** When `app.py` calls both `process()` and `process_websocket()` for the same MIDI event, it passes both calls the same `axis_positions` snapshot (one `AxisDiscovery.axes` read per event-loop tick, not re-read between the two calls) — avoiding a scenario where Cycle's axis count and, say, a fader's clamping calculation (`MAP-BANK-008`/`009`) disagree about the axis list within a single tick.

## Trigger Semantics

- **Continuous absolute** (faders, knobs): every distinct MIDI value produces an OSC send, whether the target is an OSC encoder channel, a fader's direct-axis `gotoPosition`, or a knob's bank-derived `stepPosition`. No debounce — Dragonframe needs frequent updates to drive smooth axis motion.
- **Press edge** (buttons, resets, transport, Scene): fires once when the source transitions into "pressed" (velocity/value crossing above 0, or note-on) — reused edge-detection logic from the prototype's `_evaluate`. Debounced at 80ms to absorb switch bounce, matching the prototype's default. Applies identically whether the fired output is an OSC message (`process()`) or a WebSocket command (`process_websocket()`, see WebSocket-Targeted Controls above) — the edge-detection/debounce mechanism doesn't know or care which output type a key currently produces.
- **Directional step** (jog wheel): fires once per qualifying message (nonzero sign-magnitude value), independent of any prior message — no debounce, no dedup. See Jog Wheel Frame Stepping above.

## State

The engine keeps a small per-control "previous normalized value" map, needed for both press-edge detection and continuous-value dedup (skip a resend of the identical value, regardless of whether the target is an OSC encoder channel or a direct axis). This is the only state it holds; it does not know about MIDI connection status or OSC delivery success.

- **Match invariant (CC-sourced controls only):** for every table entry whose MIDI source is a CC message (faders, knobs, Mute, Solo, transport), channel is part of the match condition exactly like the CC number — a message on any channel other than 16 fails to match and is dropped through the same "no match" path as any other non-matching event. **This invariant explicitly excludes the Scene button.** The Scene button's event carries a `channel` field decoded from its Native Mode SysEx payload (the controller's own configured global-channel ID, set during the Native Mode handshake — see `midi-input.md`), which is unrelated to "MIDI channel 16" and may legitimately be any value 0–15. The Scene button entry matches on event `type == "korg_scene"` alone, with no channel check — applying the channel-16 filter to it would silently break the Scene→Black mapping on any controller not configured to global channel 16.
- **State scope:** the previous-value map only has entries for controls that appear in the table. Unmapped/disabled controls never reach stateful evaluation at all, so no state is ever allocated for them.
- **Initial state:** every mapped control starts as "not pressed" / below threshold. A physical control already held down at app launch will not fire its first press-edge until it is released and pressed again — an accepted limitation, not specially handled.
- **Debounce semantics:** a second press-edge arriving inside the 80ms debounce window is dropped, not queued or deferred, matching the prototype's `last_sent`-gate behavior.
- **Reconnect hygiene:** the entire previous-value map is cleared whenever the MIDI Input Adapter completes a fresh connect (see `midi-input.md`), so no state survives a disconnect/reconnect cycle. `_cycle_index` (WebSocket-Targeted Controls above) is cleared the same way, reset to `-1`.

## Decisions & Alternatives

| Decision | Chosen | Alternatives Considered | Rationale |
|---|---|---|---|
| Map storage | Editable table, persisted via Preset Store | Static in-code data structure (the original design) | Reversed per the HLD: confirming/correcting a control's assignment was a real, immediate need that a fixed table couldn't satisfy |
| Continuous axis addressing | Direct axis-name addressing (`gotoPosition`), scoped to faders in Phase 1 | A virtual gamepad (blocked by macOS's DriverKit entitlement requirement — see `docs/high-level-design.md § Key Design Decisions`) | Extends the already-working OSC path instead of adding a new OS-level dependency; empirically validated against a real Dragonframe instance |
| Axis scaling range | User-specified min/max | Attempt to discover the axis's real range | Dragonframe has no `getLimits` over OSC — the range genuinely cannot be read back |
| Manual-function requirement | Documented as user-facing setup guidance, not enforced by DragonMIDI | Attempt to detect/validate the axis's Function before sending | Function type isn't exposed over OSC in any response; nothing to check against |
| Axis name entry | Picker restricted to the discovered list, no free text | Allow typing an arbitrary axis name | A typo'd free-text name would silently fail with no detection possible; the discovery mechanism already exists |
| Min/max validation | None — any real min/max pair is accepted | Reject `min > max` or `min == max` | The linear interpolation formula is well-defined either way; both are legitimate (reversed mapping, constant output) |
| Stale axis reference handling | Accepted, undetected gap — engine keeps sending to a name Dragonframe may no longer recognize | Compare against the current discovered list and warn/disable | Matches the project's pattern of accepting rare, self-evident gaps over building detection machinery |
| Target-type switch behavior | Discard the previous target's configuration entirely | Preserve/hide it for a quick switch-back | Matches "one target per entry, not multiple"; avoids stale hidden state persisting in a mapping entry or a saved preset |
| Reverting axis → encoder | `clear_axis_target(key)` fully removes the axis entry; engine falls back to the opinionated encoder target | Preserve the axis config for a later switch back to it | Symmetric with the switch-discards-state decision above; a fader has exactly one opinionated encoder target to fall back to, so there's nothing else to remember |
| Unmapped controls | Silently produce no OSC | Log "unmapped control pressed" | Keeps behavior identical to the prototype's default (disabled = no OSC); avoids noise for controls with no assigned meaning yet |
| Fader/knob update rate | Send on every distinct value, no debounce | Fixed-rate throttling (e.g. max 30/sec) | Matches prototype behavior; Dragonframe's own OSC handling is the throttle point if one is ever needed |
| Engine statefulness | Minimal (previous-value map only) | Fully stateless (push edge-detection into MIDI-IN) | Keeps MIDI-IN a pure protocol adapter; edge-detection is inherently a mapping-semantics concern, not a MIDI-parsing concern |
| Channel matching | Channel is one field of the match condition, same mechanism as CC number | Separate explicit channel-filter step | No new mechanism needed; a channel-16-only invariant falls out of the existing per-control match |
| Debounce collision handling | Drop the second press-edge inside the window | Queue and re-fire after the window closes | Matches standard debounce semantics and the prototype's existing gate behavior |
| Stuck-control initial state | Assume "not pressed" at launch; accept the one-cycle detection gap | Query controller for current state at connect | nanoKONTROL Studio has no simple state-dump facility; gap is rare and self-correcting |
| Transport/marker/track section defaults | Rewind, Previous Marker, Previous Track all → `stepBackward`; Fast Forward, Next Marker, Next Track all → `stepForward`; Cycle → `loop` | Leave Rewind/Marker/Track unmapped (no exact Dragonframe equivalent); force each onto a distinct but unrelated command | Dragonframe has no marker or track concept over OSC, but every one of these physical controls is a "step back" or "step forward" gesture — mapping all of them to the same semantic action is a better default than leaving them silent or inventing an unrelated meaning per button |
| Set Marker default | Unmapped (falls through to no-OSC) | Keep the prototype's `/dragonframe/delete` binding | That binding was an arbitrary leftover with no semantic relationship to "set marker," and a marker button silently deleting a frame is an actively dangerous default |
| Fader default target | OSC axis (direct), no name selected | OSC encoder (the original default) | Axis addressing is the primary interaction model for this project; OSC encoder remains available as a manual fallback, not the default |
| Fader-in-axis-mode, no name chosen | Send nothing | Fall back to the opinionated encoder target until configured | An axis name has no meaningful placeholder default; falling back would silently substitute a different target type than what's displayed |
| Bank derivation ownership | Engine-level: only the fader is configurable, Knob/Mute derive automatically (Solo is WebSocket-targeted, not bank-derived — see below) | Give Knob/Mute their own independent target pickers | Matches "select the bank once" — configuring one control per bank instead of three |
| Knob's derived `stepPosition` delta | `raw_value - previous_raw_value` (change since the knob's own last reading) | `raw_value - 64` (distance from center), resent on every event | Distance-from-center is not a relative step; resending it on every intermediate MIDI message the knob produces while turning caused runaway, exaggerated movement instead of a proportional nudge |
| Knob's derived `stepPosition` scale | Fixed constant `0.1` axis-position units per raw-value increment, not user-configurable | Unscaled raw delta (`1.0` per increment); user-specified sensitivity, like the fader's min/max | The unscaled raw delta moved a whole unit per MIDI tick — too coarse for the fine-adjustment purpose of the knob. A fixed constant is simpler than a configurable field until real usage shows `0.1` itself needs tuning |
| Knob's first reading after a baseline reset | Establishes the baseline silently, sends nothing | Send a delta computed against an assumed center or zero | There is no meaningful "previous reading" yet to diff against; inventing one would produce an arbitrary, not-actually-relative first move |
| Bank membership | Fader + Knob + Mute + Solo; Record/Select excluded | Include Record/Select with an invented per-axis action | No matching per-axis OSC action exists for them; matches the established "no forced unrelated mapping" pattern (Set Marker, above) |
| Mute's derived action | Mute → `setZero` — calibration (redefine the reference point to current position), not movement | Leave Mute as a static encoder reset, only Fader/Knob become axis-aware; or unmap it once an axis is assigned, since there's no "move to stored zero" command to offer instead | Gives Mute a real, distinct per-axis action rather than an arbitrary encoder-reset; accepted despite the accidental-recalibration risk since no better argument-free per-axis command exists. (Solo's bank-derived `setHome` was superseded — see the Stop/Cycle/Solo/Marker WebSocket rows below.) |
| Jog wheel target | Frame-by-frame timeline stepping (`stepForward`/`stepBackward`), direction from the sign-magnitude sign, one step per message regardless of magnitude | Keep driving `/dragonframe/encoder/17` (relative), the prototype's original binding; or scale step count to the message's magnitude | The jog wheel controlling a motion-control axis was never wanted, but leaving it silent wasted a physical control well-suited to reviewing a shot frame-by-frame. `stepForward`/`stepBackward` are already used identically by other transport controls in this map. Magnitude-scaled stepping was considered and rejected — it would require bursting several OSC sends per MIDI message for a feel not yet confirmed necessary |
| Return to Zero | Unmapped | Give it a new binding now that its original purpose (resetting the jog wheel's old encoder channel 17) no longer applies | Scoped out of this change; the jog wheel's new behavior doesn't imply any particular new meaning for Return to Zero, so it's deferred rather than guessed at |
| Jog wheel's dual OSC + keystroke output | Both fire unconditionally on every qualifying message, with no attempt to detect which one would actually have an effect | Detect whether the main timeline or Arc Motion Control workspace has focus and send only the applicable one | Matches the HLD's decision not to build frontmost/focus-detection machinery; the inapplicable output is assumed harmless (a no-op) in the context where it doesn't apply |
| Knob's cumulative nudges | Clamped to the fader's configured `[min, max]` range | Unclamped — trust Dragonframe or the user not to push it out of range | The fader defines the axis's usable range; a knob that could push past it with no floor/ceiling would defeat the purpose of that range |
| Clamp reference position | Prefer Dragonframe's live-reported axis position (`AxisDiscovery.axes`) over DragonMIDI's own internally-accumulated estimate | Track position purely from DragonMIDI's own sends, never consult live broadcasts | Dragonframe's own report is authoritative and self-correcting if reality ever diverges from the internal estimate (e.g. another client moved the axis); the internal estimate remains only as a fallback for when no live reading exists yet |
| Stop/Cycle/Solo/Marker WebSocket targets | Replace the prior OSC default entirely | Fire both OSC and WebSocket, like the jog wheel's dual output | User's explicit choice (2026-07-20) — some pairs don't make semantic sense fired together (e.g. `/dragonframe/live` + `E-Stop`); matches "one target per entry" more cleanly than the jog wheel's special-cased exception |
| Solo N's target | Unconditional `select-AXN`, removed from bank derivation | Keep Solo bank-derived (`setHome`) as before, add WebSocket only where no axis is assigned | `select-AXn` addresses Dragonframe's own WebSocket-side axis identity, unrelated to whether DragonMIDI has an OSC axis name assigned to that bank's fader — conditioning it on bank state would be a distinction without a reason |
| Cycle's axis count source | Reuse `AxisDiscovery.axes` (OSC-side discovery), not the WebSocket connection's own `replaceInputList` | Parse `replaceInputList` in the WebSocket adapter for an authoritative count | Keeps `docs/llds/websocket-output.md`'s `WS-CONN-006` (no message parsing) intact; accepts an unverified but empirically-consistent assumption that WebSocket `AXn` numbering matches OSC discovery order, rather than reopening an already-approved LLD |
| Marker ◄/► mechanism | `Jog All` (generic, targets Dragonframe's server-side "currently selected" axis) | `jog-AXn` (per-axis, as originally proposed in `docs/nanokontrol-mapping-proposal.md`) | `Jog All` is log-verified confirmed; `jog-AXn` was never tested, only inferred by naming analogy. `Jog All` also avoids DragonMIDI needing to track "current axis" itself, since Dragonframe already does after a `select-AXn` send |
| Cycle/Solo coupling | Independent — Cycle's index doesn't track Solo's last press, or vice versa | Cross-link them so Cycle continues from wherever Solo last selected | Simpler, no shared state between two otherwise-independent controls; accepted trade-off is Cycle's next target may not reflect Solo's last selection |

## Open Questions & Future Decisions

### Resolved
1. Channel matching, unmapped-control diagnostics (silent), debounce-collision handling (drop), stuck-control initial state (accepted gap), previous-value map scope (mapped controls only), and reconnect state hygiene (cleared on connect) — see Decisions & Alternatives above.
2. Editable map storage, direct axis-name addressing as the continuous-jog mechanism, user-specified axis scaling, and the Manual-function precondition — see Decisions & Alternatives above.
3. Axis name entry (picker-only), min/max validation (none needed), and stale axis reference handling (accepted gap) — see Decisions & Alternatives above.
4. Cross-spec edge audit (Phase 4): the picker-only restriction is selection-time-only, not a continuously re-enforced invariant (clarified above, consistent with the accepted-gap decision); Rescan is the mechanism for picking up newly added axes; an empty picker during the startup discovery window is acceptable, not a bug; and target-type switching discards the previous target's configuration — see the bullets above and Decisions & Alternatives.
5. The picker-only axis-selection UI, how the mapping view surfaces "never discovered yet" vs. "discovered, zero axes," and the discard-on-target-switch behavior are implemented in `docs/llds/app-ui.md`'s Mapping View section (`UI-MAP-003` through `UI-MAP-008`).
6. Fader default target (OSC axis, not encoder), fader-in-axis-mode-with-no-name send behavior (nothing, no fallback), bank derivation ownership and membership, and the knob's derived `stepPosition` scale — see Decisions & Alternatives above.
7. The jog wheel drives frame-by-frame timeline stepping; Return to Zero remains unmapped — see Decisions & Alternatives above.
8. Cross-spec edge audit (Phase 4): "one step per message" is taken to mean one step per physical detent — confirmed as a design assumption (each detent turn produces exactly one CC110 message), not independently re-derived from a hardware spec. If real-world testing on the physical wheel shows the encoder emits more than one message per felt detent, revisit with a "messages per step" divisor rather than assuming the current 1:1 mapping is correct.
9. The jog wheel additionally drives Arc Motion Control's frame stepping via a synthesized keystroke (`process_keystroke()`, `docs/llds/keystroke-output.md`), since that action has no OSC equivalent — both outputs fire unconditionally, with no frontmost/focus detection — see Decisions & Alternatives above.

### Deferred
1. Whether DragonMIDI should ever detect a non-responsive axis (e.g., by comparing sent `gotoPosition` values against subsequent position broadcasts, to warn the user their axis might be `Function: Normal` with no hardware attached) — not in scope for Phase 1's minimal implementation.
2. Exact preset persistence format (JSON schema, file location) — a code-level decision, not yet made.
3. Whether `gotoPosition`/`stepPosition` work against an axis with real, genuinely-connected motion-control hardware (`Function: Normal`, a physical device actually attached) is untested — only the "`Function: Normal`, no device attached" case has been confirmed (`docs/dragonframe-messages-research.md`). If direct addressing does not work against real connected hardware, OSC encoder channels may be more than a manual fallback for rigs using real motors, not merely a legacy option.
4. Whether Dragonframe's WebSocket-side `AXn` numbering actually corresponds, in order, to `getAllPosition`'s axis discovery order is unverified beyond both reporting the same two axes in this project's own testing — Cycle's axis-count derivation (WebSocket-Targeted Controls above) rests on this holding generally. Revisit if a project with more axes, or axes added/removed mid-session, shows Cycle selecting the wrong axis.
5. Whether Cycle and Solo should ever be cross-linked (Cycle continuing from Solo's last selection) is deferred — see the Decisions & Alternatives row above.
