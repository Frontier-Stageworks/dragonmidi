# Mapping Engine

## Context and Design Philosophy

This component is the one place MIDI meaning becomes Dragonframe meaning. Each mapping entry translates a normalized MIDI event into zero-or-one Dragonframe OSC message, through a single interface (`event -> Optional[OscMessage]`) that keeps this component swappable without touching MIDI I/O, OSC I/O, or the UI.

A second, separate interface, `process_keystroke(event) -> Optional[KeyCombo]`, exists alongside `process()` for the small number of entries whose Dragonframe function has no OSC equivalent at all (`docs/llds/keystroke-output.md`). It is not a replacement for `process()` and does not change `process()`'s return type or existing callers — the two methods are called independently, for the same event, by whatever drives the event loop (`app.py`). In this phase, the jog wheel is the only entry that produces anything from `process_keystroke()`; every other event returns `None` from it. This is a hard-coded default-map behavior, not a user-configurable target choice, so it doesn't conflict with the HLD's "one target per mapping entry" decision, which governs the general editable target-type model (Phase 2), not this phase's opinionated jog wheel default.

A third interface, `process_websocket(event, now, axis_positions) -> Optional[WebSocketCommand]`, exists for the small number of entries whose Dragonframe function is reachable only through the WebSocket output path (`docs/llds/websocket-output.md`) — Stop, Cycle, Solo 1–8, and Marker ◄/►. Unlike the jog wheel's keystroke output, these controls' WebSocket target **replaces** their prior OSC target rather than firing alongside it — each entry has exactly one target, still matching the HLD's "one target per mapping entry" principle; it's simply a WebSocket target instead of an OSC one for these four. See WebSocket-Targeted Controls below.

Unlike the original static-table design, entries are now editable (add/edit/remove, enable/disable, MIDI-learn) and persisted through a Preset Store, per `docs/high-level-design.md`. The table ships pre-loaded with the currently-selected `ControllerProfile`'s opinionated default map (`docs/llds/midi-input.md § Controller Profiles`) — the nanoKONTROL Studio's map mirrors the prior prototype's default map; the nanoKONTROL2's is a second, smaller map (see The nanoKONTROL2 Default Map below) — so a user who never opens the mapping view sees identical behavior to a fixed table, for whichever controller they've selected.

**Opinionated maps are synthesized from config, not hardcoded per profile** (`docs/llds/midi-input.md § Controller Profile Loading`). `OPINIONATED_MAP_STUDIO`/`OPINIONATED_MAP_NANOKONTROL2` are no longer literal dicts assembled from parameterless helper calls; the shared helpers (`_fader_entries()`, `_knob_entries()`, `_mute_entries()`, `_shared_button_entries()`) instead take each control's CC numbers as arguments, sourced from the active profile's config file (see Controller Profile Config Schema below). **Migration invariant:** the two bundled config files (one per existing controller) must synthesize opinionated maps byte-identical to today's two constants — this is a regression bar for this change, not an opportunity to alter either controller's default behavior.

**Profile-driven, not device-specific.** Everywhere this LLD previously hardcoded a Studio-only fact — channel 16 as the match channel, the jog wheel's CC110 dispatch, the Scene button's `korg_scene` dispatch — that fact now comes from the active `ControllerProfile` (`default_channel`, `has_jog_wheel`, `has_scene_button`). Switching the Controller Profile dropdown (`docs/llds/app-ui.md`) swaps `MappingEngine`'s active `OPINIONATED_MAP` and clears all engine state **immediately**, independent of whether a matching device has yet been found under the new profile (`docs/llds/midi-input.md § Controller Profiles`) — the previous profile's per-control state (previous values, `_cycle_index`) is meaningless against the new profile's different control set and channel, so there's no reason to wait for a physical reconnect before clearing it. The existing `on_reset_mapping`-on-fresh-connect hook (`docs/llds/midi-input.md`'s Reconnect invariant) still fires independently whenever a device actually connects under the new profile — redundant with the immediate clear, but harmless.

**Phase scope** (per `docs/high-level-design.md § Delivery Phasing`): this LLD describes the target design. **App Delivery Phase 1** implements the "OSC axis (direct)" target type, as the **default** for the 8 faders, plus **bank derivation** — knobs and the Mute button automatically follow their bank's fader assignment rather than being independently configurable (see Bank Derivation below); Solo is WebSocket-targeted and independent of bank state (see WebSocket-Targeted Controls below). Transport/marker/track buttons and Record/Select are unaffected and keep their existing targets. The jog wheel drives frame-by-frame timeline stepping (see Jog Wheel Frame Stepping below); Return to Zero remains unmapped (see the Opinionated Default Map below). The general editor (MIDI-learn, add/remove, presets, independently retargeting a knob/button) is **App Delivery Phase 2**. **App Delivery Phase 6** (this revision) adds the Group dimension — see Group Switching below — reworking Bank Derivation's axis-target storage from one value per Bank to one value per (Bank, Group), repurposing Previous/Next Track for Group stepping instead of `stepBackward`/`stepForward`, making Solo's WebSocket target Group-aware, and introducing the Preset Store's first real implementation (see Preset Store below).

## Controller Profile Config Schema (`controls:`)

Continues `docs/llds/midi-input.md § Controller Profile Config Schema`, which owns the surrounding profile-identity fields (`name`, `match_substring`, `has_native_mode`, `default_channel`, `has_jog_wheel`, `has_scene_button`, `setup_hint`). This section owns the `controls:` block's shape — the input to the shared entry-builder helpers above:

```yaml
controls:
  faders: [0, 1, 2, 3, 4, 5, 6, 7]          # 8 CC numbers, index 0 = Fader 1
  knobs:  [16, 17, 18, 19, 20, 21, 22, 23]  # 8 CC numbers, index 0 = Knob 1
  mutes:  [48, 49, 50, 51, 52, 53, 54, 55]  # 8 CC numbers, index 0 = Mute 1
  solos:  [32, 33, 34, 35, 36, 37, 38, 39]  # 8 CC numbers, index 0 = Solo 1 (WebSocket-targeted)
  transport:                                 # each key optional; omit = control doesn't exist on this device
    record: 45
    play: 41
    stop: 42            # WebSocket-targeted
    rewind: 43
    fast_forward: 44
    cycle: 46           # WebSocket-targeted
    previous_marker: 61 # WebSocket-targeted
    next_marker: 62     # WebSocket-targeted
    previous_track: 58
    next_track: 59
  jog_wheel: null        # CC number; required if has_jog_wheel is true, omitted/null otherwise
                          # (scene_button has no CC field - it's SysEx-derived, gated on has_scene_button alone)
```

- **`faders`/`knobs`/`mutes`/`solos` are fixed-length 8-element lists**, matching this LLD's Bank Derivation model of exactly 8 channel strips (Bank N = Fader N + Knob N + Mute N + Solo N). **This schema does not support a controller with a different number of channel strips** — a 4-strip or 16-strip device is out of scope for config-only onboarding in this phase; it would require changes to the bank-derivation logic itself (`FADER_KEYS`, `_KNOB_BANK_OFFSET`, etc.), not just a config file. Accepted scope boundary, not silently assumed: a pluggable profile today can vary *which* CC numbers a nanoKONTROL-shaped device uses, not the shape itself.
- **`transport` keys are individually optional.** A key's absence means that physical control doesn't exist on this device (or exists but isn't given a default mapping) — the resulting row simply isn't in the opinionated map, the same "doesn't exist as a row, not a disabled placeholder" treatment `docs/high-level-design.md` already establishes for the nanoKONTROL2's missing jog-wheel/Scene rows.
- **`solos` and the WebSocket-targeted `transport` entries (`stop`, `cycle`, `previous_marker`, `next_marker`) feed `process_websocket()`'s key-matching, not `OPINIONATED_MAP`.** These four/twelve keys are deliberately absent from the synthesized opinionated map (per `MAP-WS-009`) but still need their CC numbers from config — today's hardcoded module-level constants (`_STOP_KEY`, `_CYCLE_KEY`, `_SOLO_KEYS`, etc., all identical across both existing profiles) become profile-sourced the same way the opinionated-map helpers do.
- **`jog_wheel` is a single CC number** (today `110` for both existing profiles) rather than a list, since there's exactly one per device. Required when `has_jog_wheel: true`; the loader treats a missing value as a config error for that file (skipped — see `docs/llds/midi-input.md § Controller Profile Loading`), not a silent no-jog-wheel fallback, since that combination is self-contradictory.
- **The Scene button has no CC field.** Unlike every other control here, it's recognized from a SysEx byte pattern, not a CC number (`docs/llds/midi-input.md`'s Normalized MIDI Event) — its presence is governed entirely by `has_scene_button`, and (per that LLD's accepted-assumption note) any controller declaring it is assumed to emit the exact same hardcoded 13-byte pattern the Studio does.

## Target Types

Each entry has exactly one target, chosen by the user (or left at its default):

- **OSC action** — one of Dragonframe's fixed named commands (`shoot`, `play`, `live`, `mute`, `black`, `delete`, `shootVideoAssist`, and further commands not yet defaulted to any control).
- **OSC encoder channel** — `/dragonframe/encoder/{n}` / `/dragonframe/encoderReset/{n}`, requiring the user to separately wire that channel to an axis inside Dragonframe's Arc workspace. The fallback for a fader/knob/Mute whose bank has no axis assigned; no longer the default for faders. Solo has no OSC encoder fallback — it is WebSocket-only, unconditionally (see WebSocket-Targeted Controls below).
- **OSC axis (direct)** — addresses a discovered axis by name directly (see below). **The default target type for faders in App Delivery Phase 1.** Knobs and Mute reach this same mechanism only through bank derivation, not as an independently selectable target.
- **Custom OSC path** — arbitrary address/argument, restoring the old prototype's escape hatch. Not implemented before App Delivery Phase 2.
- **WebSocket** — one of Dragonframe's WebSocket-recognized commands, sent via `docs/llds/websocket-output.md`'s adapter. Hard-coded to Stop, Cycle, Solo 1–8, and Marker ◄/► in this phase, not independently user-selectable — see WebSocket-Targeted Controls below.

## Fader Axis Mode (Default) vs. OSC Encoder Mode

**As of 2026-07-23, this is a single engine-wide flag governing all 8 faders together, reversing the pre-existing per-fader model.** The Configuration Dialog (`docs/llds/app-ui.md § Configuration Dialog`) collapses the fader's mode control to one summary row rather than 8 independent rows, and a single flag is both simpler to reason about and directly matches that one-row UI — see Decisions & Alternatives. Mixing modes across faders (e.g. Fader 1 in axis mode, Fader 3 in encoder mode) is no longer possible; this capability was never exercised by either bundled Controller Profile's default map and is accepted as given up.

- `MappingEngine` tracks `_global_encoder_mode: bool`, defaulting `False` (**axis mode**, matching `MAP-AXIS-008`), replacing the pre-2026-07-23 per-fader `_encoder_mode: set[_Key]`. `is_axis_mode()` takes no key argument and reflects this one flag for every fader alike.
- **Every fader starts in axis mode, with no axis name selected.** In this state, moving any fader produces **no OSC output at all** for that fader — there is no fallback to the opinionated encoder target while unconfigured. An axis name is inherently project-specific; unlike an encoder channel number, there is no meaningful placeholder to fall back to until the user configures one.
- **Switching the single mode control to OSC encoder** enters encoder mode for all 8 faders simultaneously: every fader (and its bank, see Bank Derivation below) behaves exactly as documented in the Opinionated Default Map table's encoder-fallback column, regardless of what any individual fader's `_group_axis_targets` entries currently hold.
- **Switching back to OSC axis** re-enters axis mode for all 8 faders at once. **The stored (Bank, Group) axis assignments are never touched by a mode transition in either direction** — a fader whose Group 2 assignment was "PAN (0-100)" before a flip to encoder mode and back still shows "PAN (0-100)" once axis mode is restored, with no re-entry needed. This is a deliberate reversal of the pre-2026-07-23 per-fader behavior, where entering encoder mode (`clear_axis_target`) discarded that one fader's assignments across all 5 Groups outright — see Decisions & Alternatives for why that discard-on-switch rule doesn't carry over to a single switch governing all 8 faders at once.
- **The engine-level API is a single `set_fader_mode(axis: bool)` method** (replacing the pre-2026-07-23 pair `enter_axis_mode(key)` / `clear_axis_target(key)`), a no-op if the requested mode already matches the current one. On an actual transition (either direction), it discards `_previous_value`/`_axis_position` dedup/position state and the paired knob's dedup state (`MAP-BANK-007`'s mechanism) for **every** Bank's fader and knob at once, matching `_step_active_group`'s existing "discard across every affected Bank in one motion" precedent (`MAP-GROUP-011`) rather than looping a per-fader discard 8 times. It does not touch `_group_axis_targets` at all — see above.
- This is tracked by `MappingEngine` itself, not only the Configuration Dialog UI — it changes actual dispatch behavior (whether an event produces OSC output at all), not merely what a dialog displays. `set_axis_target`/`clear_group_axis_target` (per-(Bank,Group) cell editing, unchanged, see Group Switching below) are independent of this flag and do not implicitly change it.
- **Applies immediately, no Save/Cancel step** — flipping the Configuration Dialog's switch calls `set_fader_mode` directly, matching every other live-tracking control in this app (the axis pickers, the Controller Profile dropdown). There is nothing to apply or discard on dialog close.
- **A flip mid-gesture (a fader actively being moved when the switch changes) takes effect immediately, not deferred until the fader is released** — the very next MIDI value from an in-progress move dispatches under the new mode. Matches the existing "a target/mode change takes effect the instant it happens" precedent already established for mid-gesture Group switches (see Group Switching § Track ◄/► Dispatch) and for target-type switches in general; this app has no release-detection mechanism anywhere, and a mode flip doesn't introduce the first one.

## Bank Derivation

The 8 channel strips are grouped into **banks**. Bank N = Fader N, Knob N, Mute N. Solo N is grouped with the same 8 channel strips physically, but is **not** bank-derived — it always sends its WebSocket target regardless of Fader N's axis state (see WebSocket-Targeted Controls below); it is called out separately here rather than folded into "Bank N" to avoid implying its behavior depends on Fader N, the way Knob N's and Mute N's do. Record N and Select N are not part of a bank — no per-axis OSC action exists for them, so they remain unmapped regardless of bank state, unchanged from today.

**As of Phase 6, "Fader N has a real axis name assigned" means assigned *for the currently active Group*.** Bank Derivation's mechanics below are unchanged; what changed is which axis assignment they read. See Group Switching below for the (Bank, Group) storage model this section now reads from — `axis_target` in every bullet below means `self._group_axis_targets.get((fader_key, self._active_group))`, not a single fixed value per Bank.

- **Only Fader N's target is directly configurable** — the OSC axis (direct) picker described above and in `docs/llds/app-ui.md`, now one picker per Group rather than one picker total (`docs/llds/app-ui.md`'s Mapping View). Knob N and Mute N have no independent target selection; their effective target is derived from Fader N's current state for the active Group, recomputed on every event they produce, not fixed at the moment Fader N's axis was chosen or at the moment the Group last changed.
- **Fader N has a real axis name assigned (in the active Group):**
  - Knob N sends `/dragonframe/axis/{axisname}/stepPosition,f (delta)` on every distinct value once a prior reading exists to compare against, where `delta = (raw_value - previous_raw_value) * 0.1` — the change since Knob N's own last reported value, scaled down by a fixed factor of `0.1` axis-position units per MIDI raw-value increment, for fine adjustment rather than a whole-unit-per-tick jump. The first reading after Fader N's axis is assigned (or after any encoder↔axis mode transition, see below) establishes this baseline only and produces no send, since there is nothing yet to compute a change against. No debounce; the scale factor is a fixed constant, not user-configurable. **Delta must be computed against the knob's own previous reading, not its distance from a fixed center.** Dragonframe accumulates each `stepPosition` onto the axis's current position; a knob's physical rotation typically produces several intermediate MIDI messages, so a formula based on absolute distance from a fixed reference would resend a large, nearly-constant value on each one — producing runaway movement rather than a proportional nudge.
  - Mute N sends `/dragonframe/axis/{axisname}/setZero` on press.
  - **`setZero` is a calibration command, not a movement command.** It redefines the axis's zero reference point to its current position; it does not move the axis to a previously-stored zero. Dragonframe's OSC surface has no command that does the latter.
- **Fader N has no axis assigned in the active Group** (default state for that (Bank, Group) cell, or the engine-wide switch is set to OSC encoder — see Fader Axis Mode above, a single flag governing every Bank and independent of Group, not per-fader and not per-cell as of 2026-07-23): Knob N and Mute N fall back to their table-listed static defaults (`/dragonframe/encoder/{9-16}`, `/dragonframe/encoderReset/{1-8}`) — unchanged from today's behavior. This is the same "axis mode, no name chosen yet" no-fallback rule `MAP-AXIS-009` already establishes for the fader itself; Group switching doesn't introduce a second, different empty-cell behavior.
- **A repeated identical raw value produces a computed delta of `0` and is not resent** — this falls directly out of the delta formula (a repeat means `raw_value == previous_raw_value`, so the raw difference — and its scaled result — is `0`), with no separate dedup check needed.
- **Knob N's stored "previous reading" spans two different value semantics** — a normalized absolute float (0.0–1.0) in encoder mode, versus a raw MIDI int in derived-axis mode. Comparing a stale value from one semantic against a fresh value from the other, right at a mode transition, would produce a meaningless delta. Assigning Fader N's first axis name (entering axis mode from encoder mode) or clearing it (leaving axis mode for encoder mode) therefore also discards Knob N's stored reading — mirroring the same "switching a target discards prior dedup state" precedent already established for the fader itself, and forcing the next knob event to re-establish a fresh baseline rather than diff against a stale, incompatible value. Picking a *different* axis name while Fader N is already in axis mode does not discard it, since the delta computation doesn't depend on which axis is targeted.
- **A Group switch discards dedup/position state for every Bank currently in axis mode, not just the active Bank's, and not encoder-mode Banks.** Since every axis-mode Bank's effective axis potentially changes simultaneously on a Track ◄/► press (each independently may or may not have an assignment in the new Group), `_active_group` changing runs the same discard `set_axis_target`/`clear_group_axis_target` already perform per-Bank when `group == self._active_group` — `_previous_value` popped for every axis-mode fader key and its paired knob key, `_axis_position` popped for every axis-mode fader key — across all axis-mode Banks at once, rather than only the Bank(s) whose new-Group assignment actually differs from the old one. **Banks in OSC encoder mode are skipped entirely** (guarded by `is_axis_mode(fader_key)`): an encoder-mode fader's target is a fixed channel number tied to its physical position, not to Group, so its dedup state has nothing to do with the switch — discarding it would be pure waste, not merely harmless. Diffing old vs. new assignments per axis-mode Bank to discard only what actually changed was considered and rejected as unneeded complexity; the cost of the coarser per-Bank discard is one lost baseline reading per axis-mode Bank (a no-op resend next event), not a user-visible glitch.
- **A Track press mid-gesture (fader actively being moved) takes effect immediately, not deferred until release.** The very next MIDI value from an in-progress fader move fires against the new Group's axis/range with no grace period. This is a deliberate choice, not an accepted gap: it matches every other "a target change takes effect the instant it happens" precedent already established in this LLD (switching a mapping entry's target type, `docs/llds/static-mapping.md`'s "Switching a mapping entry's target type discards the previous target's configuration") — nothing in this app waits for a control to be released before a reconfiguration takes effect, and Group switching doesn't introduce a new exception to that.

### Knob N's position clamping (`MAP-BANK-008`, `MAP-BANK-009`)

Since the fader defines the axis's usable range (its configured `min`/`max`), Knob N's cumulative nudges must not be able to push the axis outside that same range — otherwise "fine-tune below the fader's floor" would silently produce an out-of-range `gotoPosition`-equivalent via repeated `stepPosition` calls, with no floor at all.

- The engine tracks, per fader key, a running position estimate built from its own sends: it starts at the fader's own last `gotoPosition` value (or, if the knob is nudged before the fader has ever sent one, the fader's configured lower bound), and updates on every knob nudge that is not itself clamped away.
- Before applying a nudge, the engine consults `AxisDiscovery.axes` (threaded through as `MappingEngine.process()`'s `axis_positions` parameter) for a live reading of that axis's actual reported position. **A live reading, when present, is authoritative** and is used in place of the internal estimate — Dragonframe's own broadcast reflects reality even if some other client (or an unexpected clamp/rounding on Dragonframe's side) moved the axis independently of DragonMIDI's estimate. The internal estimate is a fallback only, used when no live reading exists yet for that axis name (e.g. "Output Motor Positions" isn't enabled in Dragonframe, or no broadcast has arrived since startup/reconnect).
- The effective bounds are `low, high = sorted((fader.min_value, fader.max_value))`, order-independent since `MAP-AXIS-002` allows `min > max`.
- If `current_position + delta` falls within `[low, high]`, the full delta is sent unchanged, and the tracked estimate advances to the new position.
- If it would fall outside, the sent delta is reduced to exactly what's needed to land on the boundary (`clamped_position = max(low, min(high, current_position + delta))`, `clamped_delta = clamped_position - current_position`); if `current_position` is already at that boundary, the reduced delta is `0` and nothing is sent at all (distinct from `MAP-BANK-005`'s "repeated raw value" no-send case, though the observable effect — no OSC message — is the same).
- **Floating-point care**: the reduced-delta path recomputes `clamped_delta` via subtraction of two position floats, which can reintroduce floating-point error even in cases that turn out not to need clamping. The common (non-clamped) path therefore uses the original, exactly-scaled `delta` value directly rather than routing it through the same subtraction, avoiding spurious precision loss on ordinary nudges.

## OSC Axis (Direct) Target (Fader — `gotoPosition`)

- **As of Phase 6, this is a per-(Bank, Group) assignment, not a single value per fader** — see Group Switching below. Everything in this section describes how *one* such assignment behaves once it's the one the active Group currently selects; it applies identically regardless of which Group a given assignment belongs to.
- The user picks an axis name from the list the OSC Listener has discovered via `getAllPosition` (see `docs/llds/osc-io.md`), plus a **min** and **max** position value.
- For a continuous-absolute MIDI source (a fader), the engine sends `/dragonframe/axis/{axisname}/gotoPosition,f (position)` on every distinct MIDI value, with no debounce — position is computed as `min + normalized_value * (max - min)`, matching the same "continuous, no debounce" handling already used for OSC encoder targets.
- Min/max are **entered by the user, not discovered** — Dragonframe has `setLimits` but no `getLimits` over OSC, so an axis's practical range cannot be read back.
- **Setup precondition, not a DragonMIDI-enforced invariant:** the target axis's Function must be `Manual` (or otherwise not require a real connected device) for `gotoPosition` to actually move it. Empirically confirmed: an axis with `Function: Normal` and a real hardware `Connect` type (e.g. ArcMoco) but no physical device attached accepts `gotoPosition` without error but never actually moves, even with Dragonframe's "Ready to Capture" mode enabled; switching to `Function: Manual` fixed this immediately. DragonMIDI has no way to detect an axis's Function type over OSC, so this is documented user-facing guidance, not something the mapping view can validate or warn about.
- **Axis name selection is picker-only, not free text, and is gated only at selection time.** The user selects from the OSC Listener's currently-discovered axis list at the moment they configure the mapping; there is no way to type an arbitrary/undiscovered name into a direct-axis mapping, and there is no ongoing re-validation afterward (see the stale-reference bullet below — the restriction is a one-time UI gate, not a continuously-enforced invariant). A free-text field would let a typo produce a mapping that silently sends `gotoPosition` to a name Dragonframe doesn't recognize, with no way for DragonMIDI to detect the mistake.
- **Rescan to pick up newly added axes.** If the user adds an axis in Dragonframe after DragonMIDI has already discovered its axis list (or after switching to a different project), the on-demand "Rescan"/refresh action (`OSC-DISCOVER-003`, `docs/llds/osc-io.md`) re-queries `getAllPosition` and updates the discoverable list without requiring a DragonMIDI restart.
- **No validation on min/max ordering.** `min + normalized_value * (max - min)` is a well-defined linear interpolation for any real `min`/`max` pair, including `min > max` (a legitimate reversed/inverted mapping) or `min == max` (a constant output regardless of fader position, which can be a deliberate way to always send one fixed value). Neither is rejected.
- **Stale axis references are an accepted, undetected gap.** If the mapped axis name stops being reported by Dragonframe (the axis was deleted, or a different project with different axes was loaded), the engine keeps sending `gotoPosition` to that name regardless — Dragonframe presumably ignores it silently. DragonMIDI does not compare a mapping's target name against the current discovered list to warn about this, matching the project's general pattern of accepting rare, self-evident gaps rather than building detection machinery for them. Rescanning (above) helps a user find new axes; it does not retroactively validate mappings made against axes that have since disappeared.
- **Switching the engine-wide fader mode does not discard any (Bank, Group) axis configuration, as of the 2026-07-23 reversal below.** Flipping to OSC encoder mode makes every fader dispatch through the opinionated `OPINIONATED_MAP` encoder-channel lookup instead of `_group_axis_targets`, but the stored assignments themselves are untouched and immediately resume driving output the moment the switch flips back to OSC axis. This supersedes the pre-2026-07-23 per-fader behavior (`MappingEngine.clear_axis_target(key)`, now replaced by `set_fader_mode`), which discarded that one fader's axis-target entries across all 5 Groups on every switch to encoder mode — appropriate when only one fader's configuration was at stake, but judged too destructive once a single switch governs all 8 faders' up-to-40 stored assignments at once (`docs/llds/app-ui.md § Configuration Dialog`). See Fader Axis Mode above and Decisions & Alternatives for the full rationale.

## Group Switching

Introduced in Phase 6 (per `docs/high-level-design.md § Delivery Phasing`). A **Group** is a switchable dimension layered on top of Bank Derivation: which axis each Bank's fader currently addresses depends on both the Bank *and* the active Group. This section owns Group state, Track ◄/► dispatch, and the (Bank, Group) axis-target storage that Bank Derivation (above) reads from. **"Bank" keeps its existing meaning throughout this document** — a fixed, positional 1-of-8 channel-strip grouping — Group is the new, orthogonal, switchable dimension (`docs/high-level-design.md § Key Design Decisions`).

### Group Keys and Track ◄/► Dispatch

- **`ControllerProfile` gains `group_keys: GroupKeys`**, a new dataclass parallel to `WebSocketKeys` (`docs/llds/midi-input.md § Controller Profiles`): `previous: _Key | None`, `next: _Key | None`, built by a new `build_group_keys(controls)` the same way `build_websocket_keys()` derives `WebSocketKeys` from `controls.transport.get("previous_track")`/`get("next_track")`.
- **`previous_track`/`next_track` are removed from `_TRANSPORT_OSC_TARGETS`** (`mapping.py`), so `_transport_entries()`/`build_opinionated_map()` no longer synthesizes `stepBackward`/`stepForward` rows for them — the same treatment `MAP-WS-009` already gives Stop/Cycle/Solo/Marker for a different reason (WebSocket-targeted instead of OSC-targeted; here, internal-state-targeted instead of OSC-targeted). `_TRANSPORT_DISPLAY_NAMES` keeps their "Previous Track"/"Next Track" labels — the Mapping View still needs a display name for these rows (`docs/llds/app-ui.md`), it just no longer sources their *behavior* from the opinionated map.
- **`process()` special-cases Track ◄/► ahead of the generic table lookup**, the same pattern already used for the Scene button and fader-axis-mode paths: matching `key == self._profile.group_keys.previous` or `.next` steps `_active_group` (via `_press_edge`'s existing debounce/edge-detection, shared with every other button entry — `MAP-DEBOUNCE-001`) and returns `None` — **no OSC message, no WebSocket command**. This is a fifth category of special-cased control, alongside the jog wheel, Scene button, fader-axis-mode faders, and WebSocket-targeted controls — unlike all four of those, it produces no output on any of the three `process*()` interfaces; its only effect is mutating engine state that a later event's dispatch will read.
- **Dispatch order is an explicit invariant, not incidental to code layout: Track ◄/► is checked immediately after the channel-match check, before the jog-wheel special-case** (which itself precedes the fader-axis-mode and bank-derived checks — see `process()`'s existing structure). This schema places no validation against a contributed config reusing the same CC for `previous_track`/`next_track` and some other control (`faders`/`knobs`/`mutes`/`jog_wheel`) — an existing, already-accepted gap (Open Questions & Future Decisions, Deferred #7). Stating Track's position in the check order resolves *which* behavior wins on such a collision (Track's Group-switch, unconditionally) without closing the underlying validation gap itself.
- **Boundary behavior: wraps, does not clamp (confirmed 2026-07-22).** Pressing Track ► at Group 5 moves to Group 1; pressing Track ◄ at Group 1 moves to Group 5. `_active_group = (_active_group % 5) + 1` for next, `((_active_group - 2) % 5) + 1` for previous (1-indexed modulo arithmetic). Consistent with Cycle's existing wraparound (`MAP-WS-003`) rather than introducing a second, different boundary convention in the same engine.
- **A profile declaring only one of `previous_track`/`next_track` gets one-directional Group switching, not a config error.** Unlike `has_jog_wheel: true` with a missing `jog_wheel` CC (self-contradictory — the flag explicitly promises a CC exists), there is no flag here promising both keys exist; they're independently-optional `transport:` entries like every other one (`docs/llds/static-mapping.md § Controller Profile Config Schema`'s "transport keys are individually optional" rule). The absent direction simply has no row and no behavior, the same "doesn't exist, not a disabled placeholder" treatment already established throughout this LLD.

### Active Group State

- **`MappingEngine.__init__` gains `_active_group: int = 1`.** Unlike `_cycle_index` (`-1` = "nothing selected yet," a genuinely empty state), Group 1 is always a valid, meaningful starting state — there is no "no Group selected" condition to represent, so the sentinel-value convention doesn't apply here.
- **Survives an ordinary MIDI reconnect** (`reset()` does not touch `_active_group`), matching how axis-target assignments themselves already survive a reconnect (`_axis_targets` — soon `_group_axis_targets`, below — is untouched by `reset()`, only cleared by `set_profile()`). A physical unplug/replug mid-session shouldn't silently jump the user back to Group 1.
- **Reset to `1` by `set_profile()`**, alongside its existing `_group_axis_targets.clear()`/`_global_encoder_mode = False`/`_axis_position.clear()` — a different profile's Banks (different CCs, potentially a different physical device entirely) make the previous profile's active Group as meaningless as its axis assignments already are.
- **Reset to `1` on app launch, unconditionally** — matching the HLD's decision that the active Group index itself is not persisted (`docs/high-level-design.md § Key Design Decisions`), only the (Bank, Group) axis-assignment table is. `MappingEngine.__init__`'s default covers this; there is no "restore last-active Group" load path to build.
- **Read via a new `active_group` read-only property**, mirroring the existing `profile` property — needed by the Mapping View's Group indicator row (`docs/llds/app-ui.md`) and by tests. No public setter: `_active_group` only changes through a legitimate Track ◄/► press, the same encapsulation `_cycle_index` already has (no direct external mutation).

### Per-(Bank, Group) Axis-Target Storage

- **`_axis_targets: dict[_Key, _AxisTarget]` becomes `_group_axis_targets: dict[tuple[_Key, int], _AxisTarget]`**, keyed by `(fader_key, group)` instead of `fader_key` alone. `group` ranges `1`–`5` (`MAP-GROUP-002`, below).
- **`set_axis_target`/`axis_target` each gain an explicit `group: int` parameter** — they no longer implicitly operate on "whichever Group is currently active." This is deliberate: the Mapping View lets a user configure all 5 Groups' assignments regardless of which Group the physical device is currently in (`docs/llds/app-ui.md`'s 5-picker-per-row grid) — editing Group 3's assignment while Group 1 is live on the device must not require switching the device to Group 3 first.
- **`set_fader_mode(axis: bool)` (2026-07-23, replacing the pre-existing per-fader `clear_axis_target(key)`/`enter_axis_mode(key)` pair) does not touch `_group_axis_targets` at all** — see Fader Axis Mode above. It is a pure mode switch, orthogonal to the per-cell storage this section owns.
- **`clear_group_axis_target(key, group)` blanks a single (fader, Group) cell** without touching `_global_encoder_mode` and without affecting any other Group's assignment for that fader — used when the user clears one picker's selection back to "no axis chosen" while the fader stays in axis mode overall and other Groups keep their own assignments. Resolves a gap the initial Phase 6 draft left open: the two operations ("the fader is in encoder mode") and ("this one Group has no assignment yet") were conflated in the pre-Phase-6 `clear_axis_target`, which no longer held once assignments became per-Group — and remain two distinct operations now that the mode question has also gone from per-fader to engine-wide.
- **The dedup/position discard (`_previous_value.pop`, `_axis_position.pop`) inside `set_axis_target`/`clear_group_axis_target` fires only when `group == self._active_group`.** Editing or clearing a *non-active* Group's cell — e.g. picking an axis for Group 3 while Group 1 is live-streaming on the device — must not disturb Group 1's in-flight dedup baseline or clamp-position estimate, since that state belongs to whatever the device is actually addressing right now, not to whichever cell the Mapping View happens to be editing. This was a real correctness gap in the initial draft, not a cosmetic one: without this guard, background configuration of other Groups could glitch a live-moving fader.
- **`Fader Axis Mode vs. OSC Encoder Mode` is independent of Group** — whichever mode `_global_encoder_mode` currently holds applies identically regardless of the active Group; only *axis-mode* assignments (which axis name, if any) vary per (Bank, Group). As of 2026-07-23 it is also independent of Bank (see Fader Axis Mode above) — previously a per-Fader flag, now a single engine-wide one — so there is exactly one mode question to answer, not 8 (pre-2026-07-23) or 40 (a hypothetical per-cell scheme, rejected from the start — see Decisions & Alternatives).
- **`_process_bank_derived` and `_process_axis_target` (mapping.py) read `self._group_axis_targets.get((fader_key, self._active_group))`** in place of the current `self._axis_targets.get(fader_key)` / `self._axis_targets.get(key)` lookups — no other change to either method's logic.

### Solo's Group-Aware Formula

- **`process_websocket()`'s Solo branch changes from `axis_number = ws_keys.solos.index(key) + 1` to `axis_number = ws_keys.solos.index(key) + 1 + 8 * (self._active_group - 1)`.** Solo N in Group *g* now addresses axis `N + 8·(g-1)` — e.g. Solo 3 in Group 2 sends `select-AX11`. Unconditional on Bank/axis state exactly as before (`MAP-WS-002`); only the offset changes, not Solo's target *type* — it remains WebSocket-only, still not Bank-derived.
- **Cycle (`MAP-WS-003`) and Marker ◄/► (`MAP-WS-004`ff.) are unaffected** — Cycle already cycles across the full discovered axis count independent of any Bank/Group state, and Marker acts on Dragonframe's own server-side "currently selected axis," set by whichever of Solo/Cycle fired last. Group switching changes what Solo's *own* presses select; it doesn't change how Cycle or Marker behave once something is selected.

### Interaction with WebSocket-Targeted Controls

Track ◄/► is **not** added to `WebSocketKeys`/the WebSocket-Targeted Controls section above — it produces no WebSocket command, unlike Stop/Cycle/Solo/Marker. It shares only the press-edge/debounce mechanism (`_press_edge`) with those controls, not their output path.

## Preset Store (Group/Axis Configuration Persistence)

The first real implementation of the `STORE` box in `docs/high-level-design.md`'s System Design diagram — previously aspirational (Phase 2 scope, never built; `docs/llds/app-ui.md`'s Deferred section explicitly notes "no preset file in this phase"). Scoped narrowly here: **only the (Bank, Group) → axis-target table**, not the general named-preset system (add/edit/remove/enable-disable/MIDI-learn presets) that remains Phase 2 territory.

- **Location:** `~/Documents/DragonMIDI/configurations/`, sibling to the existing `~/Documents/DragonMIDI/controllers/` folder (`docs/llds/midi-input.md § Controller Profile Loading`), created on first write if it doesn't exist.
- **One file per Controller Profile, named after the profile.** A profile's Bank N is a positional concept (`MAP-CONFIG-008`'s "Bank N = index N-1 in each of `faders`/`knobs`/`mutes`"), not tied to a specific CC number, so the persisted table is keyed by **profile name + Bank index (1–8) + Group index (1–5)**, not by raw CC — the same file remains valid if a profile's config file is later edited to use different CC numbers, as long as the profile name and Bank count (fixed at 8) are unchanged.
- **Format:** a JSON object, `{"bank_axes": {"<bank_index>": {"<group_index>": {"axis_name": str, "min": float, "max": float}}}}` — omitted entries mean that (Bank, Group) cell has no assignment (equivalent to axis mode with no name chosen, `MAP-AXIS-009`). Exact schema/versioning is a code-level detail; this shape is illustrative, not final.
- **Load validation: `bank_index` and `group_index` are bounds-checked (1–8, 1–5 respectively) before use, and the value's shape (`axis_name: str`, `min`/`max: float`) is checked, before any list-indexing happens.** Closes a real bug the initial draft left open: `controls.faders[bank_index - 1]` on an unvalidated `bank_index` of `0` would silently resolve to Python's last-element negative-index behavior (`faders[-1]`, i.e. Bank 8) rather than erroring on the out-of-range value that was actually in the file. Any entry that fails validation (out-of-range index, wrong type) is **skipped with a logged warning**, not applied and not treated as fatal — the rest of the file's valid entries still load. This matches this project's general "accept rare gaps without heavy validation machinery" pattern (the stale-axis-reference precedent above) while still closing the specific silent-misassignment risk, which was a correctness bug, not a stylistic gap.
- **Module boundary, mirroring the Preset Store's read/write split from `MappingEngine` state — same separation `controller_profile_loader.py` already keeps from `controller_profile.py`.** A new `dragonmidi/preset_store.py` owns all file I/O and validation (`load_group_axis_targets(configurations_dir, profile_name) -> dict`, `save_group_axis_targets(configurations_dir, profile_name, bank_axes)`), working entirely in the plain JSON-native shape (`{bank_index: {group_index: {"axis_name", "min", "max"}}}`) — it has no dependency on `MappingEngine` or its private `_AxisTarget` type, keeping it decoupled the same way the OSC/WebSocket/Keystroke output adapters are. `MappingEngine` gains two bulk methods that translate to/from that shape: `load_group_axis_targets(bank_axes)` (populate `_group_axis_targets` directly, no save side effect — this *is* `MAP-STORE-002`'s bulk-populate path) and `dump_group_axis_targets() -> dict` (the reverse, for `app.py`'s coordinator to pass to `save_group_axis_targets` after every `set_axis_target`/`clear_group_axis_target` call — not `set_fader_mode`, which never touches this shape, see Fader Axis Mode above). Both bulk methods translate Bank index ⇄ fader `_Key` via a new `ControllerProfile.bank_fader_keys: tuple[_Key, ...]` field (index 0 = Bank 1), built by `build_bank_membership()` alongside the existing `fader_keys`/`knob_to_fader`/`mute_to_fader` — needed because `fader_keys` itself is an unordered `frozenset`, unsuitable for positional Bank-index lookup. `app.py` owns the actual `~/Documents/DragonMIDI/configurations/` path resolution and wires load-on-profile-activation and save-on-every-edit, the same composition-root pattern already used for `_user_controllers_dir()`.
- **Load timing and mechanism: a dedicated bulk-populate path, not `set_axis_target` in a loop.** Read once when a profile becomes active — at app launch for the default/last-selected profile, and again on every Controller Profile dropdown switch (`docs/llds/midi-input.md § Controller Profiles`) — translating each validated, persisted Bank index back to that profile's actual fader `_Key` via `controls.faders[bank_index - 1]`, then writing directly into `_group_axis_targets` (bypassing `set_axis_target`'s save-on-call side effect entirely). **Using `set_axis_target` for bulk load was considered and rejected**: it would trigger one full-table file rewrite per loaded entry — up to 40 redundant rewrites of the very file being read, and a process crash mid-load could leave the file truncated to a partial table instead of preserving the original complete one. The bulk-populate path only reads; it never writes as a side effect of loading. A missing file (first run for that profile) is not an error — starts with an empty table, same as today's "no axis assigned yet" default.
- **Ordering guarantee: the load happens in the same synchronous handler as `set_profile()`, not scheduled, deferred, or interleaved with the UI thread.** Since `preset_store.py` is decoupled from `MappingEngine` (see the module-boundary bullet above), the load isn't literally inside `set_profile()` itself — `app.py`'s Controller Profile switch handler calls `engine.set_profile(new_profile)` immediately followed by `engine.load_group_axis_targets(preset_store.load_group_axis_targets(...))`, both synchronous plain-Python calls with no async/threading involved (only the MIDI/OSC background threads are concurrent, per `docs/llds/app-ui.md § Threading and the Qt Bridge`). The Mapping View can only observe `_group_axis_targets` after that handler has fully returned, so there is no window where it reads a table that's been cleared but not yet reloaded.
- **Save timing:** written on every `set_axis_target`/`clear_group_axis_target` call, matching the Mapping View's existing "no separate Save step" pattern (`docs/llds/app-ui.md`'s `UI-MAP-006`) — the whole per-profile table is rewritten, not diffed, matching this project's general preference for simple over clever at this write frequency (dropdown selections, not a continuous stream). **`set_fader_mode` (2026-07-23) does not trigger a write** — since it never mutates `_group_axis_targets` (see Fader Axis Mode above), there is nothing new for the Preset Store to persist; flipping the Configuration Dialog's Fader-mode switch has no effect on the on-disk configuration file.
- **Write failure (disk full, permissions) fails silently, matching the Keystroke/WebSocket output precedent** (`docs/high-level-design.md § Key Design Decisions`) — logged, not surfaced as a new status indicator, and not retried. `_group_axis_targets` in memory remains authoritative for the running session regardless of whether the write succeeded; the practical consequence of a failed write is that the session's edits won't survive the next restart, not that they stop working now.
- **The active Group index is never written here or anywhere** — per `docs/high-level-design.md`'s decision, it always starts at Group 1 (see Active Group State above).
- **A renamed or deleted Controller Profile orphans its configuration file, and this is an accepted gap, not handled.** `docs/llds/midi-input.md` has no lifecycle handling at all for a renamed/deleted profile — profiles are simply whatever's discovered at launch — so this LLD doesn't build any either. An orphaned `configurations/<name>.json` sits unused and harmless (no corruption, no crash, no silent misapplication to the wrong profile) — matching this project's established pattern of accepting rare, self-evident gaps (stale axis references, no duplicate-CC validation) rather than building detection/cleanup machinery for them. Cross-segment edge audit finding (2026-07-22).

## Jog Wheel Frame Stepping

**Only dispatched when the active `ControllerProfile.has_jog_wheel` is `True`** (currently: the Studio only — `docs/llds/midi-input.md`). For a profile with no jog wheel (the nanoKONTROL2), `MappingEngine.process()`/`process_keystroke()` skip this special-case entirely; a CC110 message could in principle arrive from some other source, but since the nanoKONTROL2's factory map never emits it, the practical effect is simply that this behavior doesn't exist for that profile — gated explicitly on the feature flag rather than relying on "CC110 happens not to appear in the other device's map," so the guard stays correct even if a future third profile's map happened to reuse CC110 for something else.

The jog wheel (CC 110) reports movement using the nanoKONTROL Studio's KORG sign-magnitude relative encoding, reused byte-for-byte from the prior prototype's decode (`relative_sign_magnitude` trigger): a message's raw value `1–63` means clockwise rotation by that magnitude, `65–127` means counterclockwise rotation by `raw_value - 64`, and `0` or `64` both mean no movement.

- **Direction only, magnitude ignored.** Every clockwise message (`1–63`) sends `/dragonframe/stepForward` exactly once; every counterclockwise message (`65–127`) sends `/dragonframe/stepBackward` exactly once. The message's magnitude is not used to send multiple steps or to scale anything — one physical detent is one frame, regardless of how fast the wheel was spun. A message of `0` or `64` produces no OSC output.
- **No debounce, no dedup.** Unlike button-type entries (`MAP-DEBOUNCE-001`'s 80ms window) or continuous-absolute entries (skip-on-repeated-value), every qualifying jog wheel message fires immediately and independently — each message already represents one discrete physical detent, not a bounce or a resend of an unchanged value. A rapid run of identical raw values (e.g. several `1`s in a row from a fast spin) is several distinct detents, not one repeated reading, and each fires its own step.
- **Stateless.** The jog wheel entry holds no per-control state (no `_previous_value`, `_pressed_state`, or `_last_fired` entry) — its OSC output is a pure function of each message's raw value. There is nothing to clear on MIDI reconnect for this control.
- **Channel match still applies.** As a CC-sourced control, the jog wheel is still subject to `MAP-TABLE-001`'s channel match invariant like every other CC entry — a jog wheel message on any channel other than the active profile's `default_channel` does not match.
- **Implementation shape:** the jog wheel's target address is conditional on the message's sign, which doesn't fit `_MapEntry`'s fixed-address shape used by the rest of `OPINIONATED_MAP`. It is special-cased in `MappingEngine.process()` ahead of the generic table lookup, the same way the Scene button and fader-axis-mode paths already are, rather than adding a second address field to `_MapEntry` for one control.

### Keystroke Output (Arc Motion Control)

**Also gated on `has_jog_wheel`** — this entire sub-section is Studio-only, since it exists purely to give the jog wheel a second output; a profile with no jog wheel has nothing to gate this behavior on in the first place.

The main timeline's `stepForward`/`stepBackward` OSC commands have no effect while the Arc Motion Control workspace is focused (`docs/high-level-design.md § Problem`). Dragonframe's only reachable equivalent there is a distinct action, "Step Moco Forward"/"Step Moco Back," with no OSC message of its own, only a Hot Key binding (`Option+Shift+Right`/`Option+Shift+Left` by default).

- On the same event and using the same sign-magnitude direction decode as the OSC output above, `MappingEngine.process_keystroke()` returns `KeyCombo(frozenset({"alt", "shift"}), "right")` for a clockwise message and `KeyCombo(frozenset({"alt", "shift"}), "left")` for a counterclockwise one; `0`/`64` and any non-jog-wheel event return `None`. `"alt"` is the cross-platform name for the physical key Dragonframe's default binding calls "Option" on macOS.
- **Both outputs fire for the same message.** A single jog wheel MIDI message produces one `process()` call and one `process_keystroke()` call from the event loop, each independently evaluated — there is no attempt to detect which of the two would actually have an effect (main timeline vs. Arc workspace focused) and suppress the other. The OSC command is a no-op when Arc has focus; the synthesized keystroke is presumably a no-op (or at least harmless) when the main timeline has focus and Arc isn't open. This mirrors `docs/high-level-design.md § Non-Goals`' decision not to build frontmost/focus-detection machinery.
- **Assumes the default Hot Key binding.** If the user has remapped "Step Moco Forward"/"Step Moco Back" away from `Option+Shift+Right`/`Left` in Dragonframe's own Hot Keys preferences, the synthesized keystroke will silently do nothing (or trigger whatever else that combo is now bound to) — DragonMIDI has no way to read Dragonframe's actual configured bindings, the same "documented precondition, not detected" treatment as the axis `Function: Manual` requirement.
- **No debounce or dedup**, matching the OSC output's `MAP-JOG-004` and `docs/llds/keystroke-output.md`'s "no debounce or dedup at this layer" — every qualifying message produces its own keystroke, independent of prior messages.
- **Stateless, like the OSC-side jog wheel handling.** `process_keystroke()` reads only the event's raw value; it neither consults nor mutates `_previous_value`/`_pressed_state`/`_last_fired`, so it needs no reconnect-clear handling and `tracked_controls()` is unaffected by it (`MAP-JOGKEY-007`).

## WebSocket-Targeted Controls

Four controls target `docs/llds/websocket-output.md`'s adapter instead of OSC, replacing their prior OSC default entirely (not firing alongside it, unlike the jog wheel's dual OSC+keystroke output above). Both the Studio and the nanoKONTROL2 have all four physical controls (Stop, Cycle, Solo 1–8, Marker ◄/►), so unlike the jog wheel/Scene sections above, nothing here is feature-gated — only the `_Key` match tuples' channel component varies per active profile's `default_channel`, the same as every other CC-sourced entry (`MAP-TABLE-001`). All four are press-edge controls and reuse the same edge-detection and 80ms debounce logic as any other button entry (`MAP-DEBOUNCE-001`) — `process_websocket()` shares `_pressed_state`/`_last_fired` state with `process()`, keyed by the same `_Key` tuples. This relies on `MAP-WS-009` (below): these four keys are removed from `OPINIONATED_MAP` entirely, so `process()` never matches them and there is no collision in reusing the same dicts across the two methods for the same key.

**Terminology note:** `process_websocket()` *returns* a `WebSocketCommand`; whether it's actually delivered to Dragonframe is `docs/llds/websocket-output.md`'s concern, not this component's. If no WebSocket connection is currently active, `WS-SEND-004` silently drops it — from this LLD's point of view, the press still "produces a command" every time; "produces" below should be read that way, not as a delivery guarantee. This is consistent with `docs/high-level-design.md`'s decision that WebSocket output fails silently with no dedicated indicator — a press with no active connection is simply not observable to the user as having done anything, the same as any other silent-fail case in this app.

- **Stop (CC 42) → `E-Stop`.** Produces the bare trigger `WebSocketCommand("E-Stop")` on press. Removed from `OPINIONATED_MAP`; `process()` no longer matches this key at all (falls through to `MAP-TABLE-005`'s no-match path).
- **Solo N (CC 32–39) → `select-AX(N + 8·(g-1))`, where `g` is the active Group.** Produces `WebSocketCommand(f"select-AX{N + 8 * (g - 1)}")` on press (Phase 6; see Group Switching below) — unconditional, independent of Bank N's fader/axis state (see Bank Derivation above) and independent of the discovered axis count (unlike Cycle, below). If the project has fewer than `N + 8·(g-1)` axes, Solo N addresses an axis identity Dragonframe doesn't have; this is accepted the same way a stale OSC axis-name reference is accepted elsewhere in this LLD — DragonMIDI doesn't validate against a live count before sending. Removed from bank derivation entirely; `_process_bank_derived` no longer has a Solo branch.
- **Cycle (CC 46) → `select-AXn`, cycling.** Produces `WebSocketCommand(f"select-AX{n}")` on press, where `n` advances by one (1-indexed) on every press and wraps modulo the current axis count. Removed from `OPINIONATED_MAP`.
  - The axis count is `len(axis_positions or {})`, reusing the same `axis_positions` parameter `process()` already receives from `AxisDiscovery.axes` (`docs/llds/osc-io.md`) — **not** a count read from the WebSocket connection's own `replaceInputList`, which `docs/llds/websocket-output.md`'s `WS-CONN-006` deliberately never parses. This rests on an **unverified assumption**: that Dragonframe's WebSocket-side `AX1`/`AX2`/… numbering corresponds, in order, to the axis names OSC's `getAllPosition` discovers. Both were confirmed to report the same two axes in this project's testing, but the correspondence itself — as opposed to each mechanism's individual behavior — was never directly tested.
  - `len(axis_positions or {})` collapses `AxisDiscovery`'s two distinct empty states (`None` = never queried; `{}` = queried, zero axes — `docs/llds/osc-io.md`) into the same `0` count. During the ~2-second discovery-timeout window right after a fresh MIDI/OSC reconnect, a Cycle press therefore behaves identically to "genuinely zero axes": no command produced. This is an accepted simplification, not a distinction worth threading through — either way, there's nothing yet to cycle to.
  - If the axis count is `0`, Cycle presses still register as a normal press edge against `_pressed_state`/`_last_fired` (the debounce window still applies to the next press) but produce no WebSocket command and do not advance `_cycle_index` — there is nothing to cycle through.
  - `MappingEngine` tracks one new piece of state, `_cycle_index: int`, starting at `-1` (nothing selected yet). Each press computes `_cycle_index = (_cycle_index + 1) % axis_count` and sends `select-AX{_cycle_index + 1}`. Cleared on MIDI reconnect, matching the rest of the engine's state (see State below).
  - **This reconnect-clear is MIDI-scoped, not WebSocket-scoped, and the two can drift.** `_cycle_index` resets whenever the MIDI Input Adapter reconnects; Dragonframe's own server-side "currently selected axis" is tied to the WebSocket connection, which can reconnect independently (e.g. Dragonframe regains OS focus with no MIDI hiccup at all — Dragonframe's focus-triggered WebSocket reconnect behavior). After either reconnect happens without the other, Cycle's next press may not match whatever axis Dragonframe still considers selected from before. **Accepted as a known limitation**, not built around — matches this project's established pattern of accepting rare, self-evident state-divergence gaps (e.g. stale OSC axis references, above) rather than adding cross-component connection-change notification machinery that neither `docs/llds/websocket-output.md` nor this LLD currently has.
  - Cycle's index is entirely independent of Solo's presses in the sense that pressing Solo N does not read or write `_cycle_index`, and vice versa — **but both write the same Dragonframe-side "currently selected axis"**, so they are not independent in effect: Solo N pressed after Cycle (or vice versa) changes what a subsequent Marker press (below) will act on. "Independent" here means "no shared DragonMIDI-side state," not "no shared effect on Dragonframe."
- **Marker ◄/► (CC 61 / CC 62) → `Jog All`.** Previous Marker produces `WebSocketCommand("Jog All", operation="+", params=(-1,))`; Next Marker produces `WebSocketCommand("Jog All", operation="+", params=(1,))`, one increment per press (matching the jog wheel's "one detent, one step" precedent, not a continuous stream). Both removed from `OPINIONATED_MAP`.
  - `Jog All` is a generic, not-axis-specific command — it acts on whichever axis Dragonframe's own server-side state currently considers "selected," last set by a `select-AXn` send from **either** Solo or Cycle (both feed the same Dragonframe-side state — see above). **Marker ◄/► therefore does nothing until Solo or Cycle has selected an axis at least once** in the current Dragonframe session; this is an accepted dependency between these controls, not a bug. DragonMIDI has no way to observe whether that prior selection actually landed (`WS-CONN-006` parses nothing back), so this dependency is inherently untestable from DragonMIDI's own state — verifiable only by observing Dragonframe itself.
  - Chosen over the originally-proposed `jog-AXn`: `Jog All` has direct log-verified confirmation, while `jog-AXn` was never tested, only inferred by naming-pattern analogy with `select-AXn`/`Step`. Using `Jog All` also means `MappingEngine` doesn't need to track "which axis is currently selected" itself — Dragonframe already does, and `select-AXn`'s confirmed retargeting behavior means DragonMIDI doesn't need to duplicate that state.

**No cross-output ordering guarantee.** `process()`'s OSC send and `process_keystroke()`'s keystroke send are synchronous on the calling thread; `process_websocket()`'s result is handed to the WebSocket adapter's `send()`, which itself hands off to a separate event-loop thread (`docs/llds/websocket-output.md § Runtime Model`, `WS-RUNTIME-001`). Nothing in this app guarantees that a WebSocket command reaches Dragonframe before or after an OSC/keystroke send for a later MIDI event — two rapid presses (e.g. Cycle then Marker) could theoretically be observed out of order at Dragonframe. Accepted: MIDI events are still drained and dispatched to all three `process*()` calls in strict arrival order on the Qt thread, and the async handoff is expected to be fast in practice; building a cross-output ordering guarantee would mean serializing all three output paths (OSC, Keystroke, WebSocket) behind one send queue, a real architectural change out of proportion to a theoretical race.

**Same axis-count snapshot within one event.** When `app.py` calls both `process()` and `process_websocket()` for the same MIDI event, it passes both calls the same `axis_positions` snapshot (one `AxisDiscovery.axes` read per event-loop tick, not re-read between the two calls) — avoiding a scenario where Cycle's axis count and, say, a fader's clamping calculation (`MAP-BANK-008`/`009`) disagree about the axis list within a single tick.

## Trigger Semantics

- **Continuous absolute** (faders, knobs): every distinct MIDI value produces an OSC send, whether the target is an OSC encoder channel, a fader's direct-axis `gotoPosition`, or a knob's bank-derived `stepPosition`. No debounce — Dragonframe needs frequent updates to drive smooth axis motion.
- **Press edge** (buttons, resets, transport, Scene): fires once when the source transitions into "pressed" (velocity/value crossing above 0, or note-on) — reused edge-detection logic from the prototype's `_evaluate`. Debounced at 80ms to absorb switch bounce, matching the prototype's default. Applies identically whether the fired output is an OSC message (`process()`) or a WebSocket command (`process_websocket()`, see WebSocket-Targeted Controls above) — the edge-detection/debounce mechanism doesn't know or care which output type a key currently produces.
- **Directional step** (jog wheel): fires once per qualifying message (nonzero sign-magnitude value), independent of any prior message — no debounce, no dedup. See Jog Wheel Frame Stepping above.

## State

The engine keeps a small per-control "previous normalized value" map, needed for both press-edge detection and continuous-value dedup (skip a resend of the identical value, regardless of whether the target is an OSC encoder channel or a direct axis). This is the only state it holds; it does not know about MIDI connection status or OSC delivery success.

- **Match invariant (CC-sourced controls only):** for every table entry whose MIDI source is a CC message (faders, knobs, Mute, Solo, transport), channel is part of the match condition exactly like the CC number — a message on any channel other than the active `ControllerProfile`'s `default_channel` (16 for the Studio, 1 for the nanoKONTROL2 — `docs/llds/midi-input.md`) fails to match and is dropped through the same "no match" path as any other non-matching event. **This invariant explicitly excludes the Scene button, when the active profile has one (`has_scene_button`).** The Scene button's event carries a `channel` field decoded from its Native Mode SysEx payload (the controller's own configured global-channel ID, set during the Native Mode handshake — see `midi-input.md`), which is unrelated to the profile's `default_channel` and may legitimately be any value 0–15. The Scene button entry matches on event `type == "korg_scene"` alone, with no channel check — applying the channel filter to it would silently break the Scene→Black mapping on any controller not configured to that channel. For a profile with no Scene button (the nanoKONTROL2), this exclusion is moot: no `korg_scene` event is ever produced (`docs/llds/midi-input.md`), so there is nothing for the entry to match regardless.
- **State scope:** the previous-value map only has entries for controls that appear in the table. Unmapped/disabled controls never reach stateful evaluation at all, so no state is ever allocated for them.
- **Initial state:** every mapped control starts as "not pressed" / below threshold. A physical control already held down at app launch will not fire its first press-edge until it is released and pressed again — an accepted limitation, not specially handled.
- **Debounce semantics:** a second press-edge arriving inside the 80ms debounce window is dropped, not queued or deferred, matching the prototype's `last_sent`-gate behavior.
- **Reconnect hygiene:** the entire previous-value map is cleared whenever the MIDI Input Adapter completes a fresh connect (see `midi-input.md`), so no state survives a disconnect/reconnect cycle. `_cycle_index` (WebSocket-Targeted Controls above) is cleared the same way, reset to `-1`. **A Controller Profile switch clears the same state immediately**, at the moment the dropdown changes — not deferred until a device under the new profile actually connects (`docs/llds/midi-input.md § Controller Profiles`) — and `MappingEngine` swaps its active `OPINIONATED_MAP` to the new profile's map in that same step, so a profile switch both clears state and changes which controls exist together, up front. The ordinary reconnect-clear above still fires independently whenever a device later connects under the (possibly new) profile, redundantly but harmlessly.
- **Group state hygiene (Phase 6):** unlike `_cycle_index`, `_active_group` is **not** cleared by an ordinary MIDI reconnect (`reset()`) — it survives a physical unplug/replug within the same session, matching how `_group_axis_targets` itself already survives a reconnect. `_active_group` **is** reset to `1` by a Controller Profile switch (`set_profile()`), alongside `_group_axis_targets.clear()` — a different profile's Banks make the previous profile's Group assignments and active index equally meaningless. See Group Switching § Active Group State for the full rationale.

| Preset Store scope | Only the (Bank, Group) axis table | The full general named-preset system (add/edit/remove, MIDI-learn, enable/disable) | The general system remains explicitly Phase 2 scope (`docs/high-level-design.md § Delivery Phasing`); building it now would be far more than this change needs |
| Orphaned configuration file on profile rename/delete (Phase 4 cross-segment audit) | Accepted gap — not detected, not cleaned up | Detect orphans and prompt to delete, or garbage-collect on a schedule | `docs/llds/midi-input.md` has no rename/delete lifecycle handling for profiles at all; matching that with new file-watching machinery here would be new complexity out of proportion to a rare, harmless (unused file, no corruption) gap |
| Preset Store load validation (edge-case probe) | Bounds-check `bank_index`/`group_index`, skip invalid entries with a logged warning | Trust the file's contents (app-managed, not user-authored) | Closes a real bug: an unvalidated `bank_index` of `0` resolves via Python negative-indexing to `faders[-1]` (Bank 8) instead of erroring — a silent misassignment, not just a skipped entry |
| Preset Store load mechanism (edge-case probe) | Dedicated bulk-populate path writing directly to `_group_axis_targets` | Call `set_axis_target` in a loop over the loaded entries | Looping `set_axis_target` would trigger up to 40 redundant full-table rewrites of the file being read, and risk truncating it on a mid-load crash |
| Preset Store load/clear ordering (edge-case probe) | `set_profile()` performs the load synchronously before returning | Defer/schedule the load separately from the clear | `MappingEngine` methods are plain synchronous calls on one thread; no real race exists once this is stated as an explicit invariant, only an undocumented assumption did |
| Preset Store write failure (edge-case probe) | Fails silently — logged, not retried, no new status indicator | Surface a dedicated persistence-failure indicator | Matches the existing Keystroke/WebSocket "fails silently" precedent (`docs/high-level-design.md`); in-memory state stays authoritative for the session regardless |
| Background-Group edit vs. live dedup state (edge-case probe) | `set_axis_target`/`clear_group_axis_target`'s dedup/position discard fires only when `group == self._active_group` | Discard unconditionally on every edit, regardless of Group | Editing a non-active Group's picker must not glitch a live-moving fader's baseline in the active Group — a real correctness bug in the initial draft, not a style choice |
| `clear_axis_target` scope, Phase 6 (superseded 2026-07-23) | *Originally:* split into `clear_axis_target(key)` (full per-fader encoder-mode toggle, all 5 Groups) and `clear_group_axis_target(key, group)` (one cell only) | Keep one method, add a `group` parameter that sometimes means "all" and sometimes means "one" | A single method silently overloading two different operations ("this fader is in encoder mode" vs. "this one Group has no assignment yet") was the actual gap; splitting them removed the ambiguity outright. `clear_axis_target` itself was later replaced by the non-destructive `set_fader_mode` (2026-07-23, see above); `clear_group_axis_target` is unaffected and still the only per-cell clear |
| Group-switch discard scope (edge-case probe) | Only axis-mode Banks, guarded by `is_axis_mode(fader_key)` | Discard every Bank unconditionally, including encoder-mode ones | An encoder-mode fader's target doesn't depend on Group at all; discarding its dedup state would be pure waste, not merely harmless, once named explicitly |
| Mid-gesture Group switch (edge-case probe) | Takes effect immediately, no deferral to release | Ignore/defer a Track press while a fader is actively moving | Matches every other "a target change takes effect the instant it happens" precedent already in this LLD; a release-detection mechanism would be new complexity this app has never needed elsewhere |
| Asymmetric `previous_track`/`next_track` config (edge-case probe) | Valid — one-directional Group switching, not a config error | Reject as invalid, mirroring the `has_jog_wheel`/missing-CC self-contradiction check | No flag here promises both keys exist, unlike `has_jog_wheel: true`; these are independently-optional `transport:` entries like every other one |
| Track vs. jog-wheel dispatch order under CC collision (edge-case probe) | Explicit invariant: Track checked immediately after the channel-match check, before the jog-wheel special-case | Leave the order unspecified/incidental to code layout | Makes the outcome of a hypothetical contributed-config CC collision deterministic and documented, without needing to close the separate, already-accepted no-duplicate-CC-validation gap |

## Open Questions & Future Decisions

### Resolved
1. Channel matching, unmapped-control diagnostics (silent), debounce-collision handling (drop), stuck-control initial state (accepted gap), previous-value map scope (mapped controls only), and reconnect state hygiene (cleared on connect) — see Decisions & Alternatives above.
2. Editable map storage, direct axis-name addressing as the continuous-jog mechanism, user-specified axis scaling, and the Manual-function precondition — see Decisions & Alternatives above.
3. Axis name entry (picker-only), min/max validation (none needed), and stale axis reference handling (accepted gap) — see Decisions & Alternatives above.
4. Cross-spec edge audit (Phase 4): the picker-only restriction is selection-time-only, not a continuously re-enforced invariant (clarified above, consistent with the accepted-gap decision); Rescan is the mechanism for picking up newly added axes; an empty picker during the startup discovery window is acceptable, not a bug; and target-type switching discards the previous target's configuration — see the bullets above and Decisions & Alternatives.
5. The picker-only axis-selection UI, how the mapping view surfaces "never discovered yet" vs. "discovered, zero axes," and the discard-on-target-switch behavior are implemented in `docs/llds/app-ui.md`'s Mapping View section (`UI-MAP-004` through `UI-MAP-008`; the encoder/axis toggle UI these once included, `UI-MAP-003`, was removed 2026-07-23 — see `UI-MAP-018`).
6. Fader default target (OSC axis, not encoder), fader-in-axis-mode-with-no-name send behavior (nothing, no fallback), bank derivation ownership and membership, and the knob's derived `stepPosition` scale — see Decisions & Alternatives above.
7. The jog wheel drives frame-by-frame timeline stepping; Return to Zero remains unmapped — see Decisions & Alternatives above.
8. Cross-spec edge audit (Phase 4): "one step per message" is taken to mean one step per physical detent — confirmed as a design assumption (each detent turn produces exactly one CC110 message), not independently re-derived from a hardware spec. If real-world testing on the physical wheel shows the encoder emits more than one message per felt detent, revisit with a "messages per step" divisor rather than assuming the current 1:1 mapping is correct.
9. The jog wheel additionally drives Arc Motion Control's frame stepping via a synthesized keystroke (`process_keystroke()`, `docs/llds/keystroke-output.md`), since that action has no OSC equivalent — both outputs fire unconditionally, with no frontmost/focus detection — see Decisions & Alternatives above.
10. **nanoKONTROL2 support:** the channel-match invariant, jog-wheel/Scene-button dispatch, and `OPINIONATED_MAP` are all sourced from the active `ControllerProfile` rather than hardcoded to the Studio; the nanoKONTROL2's own map omits jog-wheel and Scene rows entirely; a Controller Profile switch is treated as a reconnect for state-hygiene purposes — see Decisions & Alternatives above and `docs/llds/midi-input.md § Controller Profiles`.
11. **The nanoKONTROL2 Default Map's CC numbers are confirmed working against real hardware (2026-07-21)** — every control produced its expected Dragonframe behavior in practice. This was a practical confirmation (controls behaved correctly), not an independent byte-level MIDI-monitor trace of each raw CC number; if a specific number later turns out wrong under different real-world conditions, only the nanoKONTROL2 profile's map data needs correction — the profile-driven architecture (channel matching, jog-wheel/Scene gating) is independent of the specific numbers.
12. **Config-driven opinionated maps (Phase 5):** the shared entry-builder helpers become parameterized by CC numbers sourced from the active profile's `controls:` config block rather than hardcoded per-profile literals; the two existing profiles' bundled config files must reproduce today's two `OPINIONATED_MAP_*` constants byte-for-byte. WebSocket-targeted keys (Solo, Stop, Cycle, Marker ◄/►) are config-sourced the same way. The channel-strip count remains fixed at 8 — not itself config-driven — see Decisions & Alternatives above.
13. **Bank membership determination (Phase 5 edge audit, 2026-07-22):** Bank N pairing (which Knob/Mute belongs to which Fader) is computed by positional index per-profile, not fixed CC-arithmetic offsets — the pre-Phase-5 offset approach assumed `knobs`/`mutes` CCs were always arithmetically aligned with `faders`', which a config's independently-declared CC lists don't guarantee (`MAP-CONFIG-008`) — see Decisions & Alternatives above.
14. **Group switching edge-case probe (Phase 6, 2026-07-22):** Preset Store load validation and the bank-index negative-indexing bug it closes; the bulk-populate load path that avoids N redundant file rewrites; the `set_profile()`-is-synchronous ordering guarantee; write-failure handling (silent, matching Keystroke/WebSocket precedent); the `group == self._active_group` guard preventing background-Group edits from disturbing a live gesture's dedup state; the `clear_axis_target`/`clear_group_axis_target` split; scoping the Group-switch discard to axis-mode Banks only; the deliberate immediate-effect (not deferred-to-release) mid-gesture Group switch; asymmetric one-directional Track config being valid, not an error; and the explicit Track-vs-jog-wheel dispatch order — see Group Switching and Preset Store above, and the new Decisions & Alternatives rows below.
15. **Group boundary behavior confirmed (2026-07-22): wraps, does not clamp** — see Group Switching § Group Keys and Track ◄/► Dispatch.
16. **Fader Axis Mode becomes a single engine-wide flag, not per-fader (2026-07-23 reversal):** `_global_encoder_mode: bool` replaces `_encoder_mode: set[_Key]`; `set_fader_mode(axis: bool)` replaces `enter_axis_mode(key)`/`clear_axis_target(key)` and — unlike the method it replaces — never discards `_group_axis_targets`, only changes which behavior `process()` currently exhibits. Driven by the Configuration Dialog's single Fader-mode switch (`docs/llds/app-ui.md § Configuration Dialog`), which has no per-fader equivalent to control 8 independent flags with. See Fader Axis Mode above and Decisions & Alternatives.

### Deferred
1. Whether DragonMIDI should ever detect a non-responsive axis (e.g., by comparing sent `gotoPosition` values against subsequent position broadcasts, to warn the user their axis might be `Function: Normal` with no hardware attached) — not in scope for Phase 1's minimal implementation.
2. Exact preset persistence format for the *general* named-preset system (Phase 2 scope: add/edit/remove, MIDI-learn, enable/disable) remains undecided. The narrower (Bank, Group) axis-table persistence is now decided — see Preset Store above; its exact JSON schema/versioning is still illustrative, not finalized, but is a code-level detail, not an LLD-level decision.
3. Whether `gotoPosition`/`stepPosition` work against an axis with real, genuinely-connected motion-control hardware (`Function: Normal`, a physical device actually attached) is untested — only the "`Function: Normal`, no device attached" case has been confirmed. If direct addressing does not work against real connected hardware, OSC encoder channels may be more than a manual fallback for rigs using real motors, not merely a legacy option.
4. Whether Dragonframe's WebSocket-side `AXn` numbering actually corresponds, in order, to `getAllPosition`'s axis discovery order is unverified beyond both reporting the same two axes in this project's own testing — Cycle's axis-count derivation (WebSocket-Targeted Controls above) rests on this holding generally. Revisit if a project with more axes, or axes added/removed mid-session, shows Cycle selecting the wrong axis.
5. Whether Cycle and Solo should ever be cross-linked (Cycle continuing from Solo's last selection) is deferred — see the Decisions & Alternatives row above.
6. Whether a future config schema revision should support a different channel-strip count (not 8), requiring Bank Derivation's core logic to generalize — deferred until a contributed profile actually needs it (see Decisions & Alternatives above).
7. Exact validation rules for `controls:` beyond "the right keys with integer CC values" (e.g. rejecting out-of-range CC numbers, rejecting duplicate CCs reused across two different controls on the same device) are not yet specified.
