# Static Mapping Engine — EARS Specs

Traces to `docs/llds/static-mapping.md`.

## Controller Profile Behavior

- [x] **MAP-PROFILE-001**: The system shall maintain two opinionated maps, one per Controller Profile (`docs/llds/static-mapping.md § The Opinionated Default Map (nanoKONTROL Studio)` and `§ The nanoKONTROL2 Default Map`), and shall dispatch every MIDI event against whichever map corresponds to the currently-active Controller Profile (`MIDI-PROFILE-002`/`003`, see `docs/specs/midi-input.md`).
- [x] **MAP-PROFILE-002**: The nanoKONTROL2's opinionated map shall assign the same control-to-target behavior as the nanoKONTROL Studio's map for every control both devices have (faders, knobs, Mute, Solo, transport, Cycle, Marker, Track), using the same CC numbers and MIDI channel 1 (zero-indexed 0) in place of the Studio's channel 16 — the CC numbers are carried over from commonly-documented factory defaults and were confirmed working against real nanoKONTROL2 hardware on 2026-07-21 (see `docs/llds/static-mapping.md`'s Decisions & Alternatives).
- [x] **MAP-PROFILE-003**: The nanoKONTROL2's opinionated map shall contain no jog wheel entry and no Scene button entry; its unmapped-by-default set shall be Record 1–8 and Set Marker (`MAP-TABLE-005`), not the Studio's Record 1–8/Select 1–8/Set Marker/Return to Zero set, since the nanoKONTROL2 has no Select buttons and no Return to Zero control.
- [x] **MAP-PROFILE-004**: When the active Controller Profile switches, the system shall swap `MappingEngine`'s active opinionated map to the newly-selected profile's map and clear all previous-value/pressed/debounce state, `_cycle_index`, `_group_axis_targets`, the engine-wide fader mode (reset to OSC axis (direct), `MAP-AXIS-010`), and `_active_group` (reset to `1`, per `MAP-GROUP-007`) immediately (`MIDI-PROFILE-005`, see `docs/specs/midi-input.md`), independent of whether a physical device has yet been found under the new profile.

## Controller Profile Config Schema (`controls:`)

- [x] **MAP-CONFIG-001**: The system shall synthesize each Controller Profile's opinionated map from its config file's `controls:` block, via shared entry-builder logic parameterized by that profile's declared CC numbers, rather than from a hardcoded per-profile literal map.
- [x] **MAP-CONFIG-002**: The system shall accept a `controls:` block with the following structure: `faders`, `knobs`, `mutes`, and `solos`, each an 8-element list of CC numbers (index 0 = control 1); a `transport` mapping of named buttons (`record`, `play`, `stop`, `rewind`, `fast_forward`, `cycle`, `previous_marker`, `next_marker`, `previous_track`, `next_track`) to CC numbers, any of which may be omitted; and `jog_wheel`, a single CC number.
- [x] **MAP-CONFIG-003**: The bundled nanoKONTROL Studio and nanoKONTROL2 config files, once loaded through `MAP-CONFIG-001`/`002`, shall synthesize opinionated maps identical in content to this LLD's pre-Phase-5 `OPINIONATED_MAP_STUDIO` and `OPINIONATED_MAP_NANOKONTROL2` constants.
- [x] **MAP-CONFIG-004**: If a `transport` key is omitted from a Controller Profile's `controls:` block, then the system shall treat that physical control as not existing for that profile — no row shall appear for it in the opinionated map or the Mapping View, matching `MAP-PROFILE-003`'s "absent, not disabled" treatment.
- [x] **MAP-CONFIG-005**: The system shall source the WebSocket-targeted keys' CC numbers (`solos`, and `transport`'s `stop`/`cycle`/`previous_marker`/`next_marker`) from the active Controller Profile's `controls:` block, for use by `process_websocket()`'s key matching (`MAP-WS-001` through `MAP-WS-009`), the same way `MAP-CONFIG-001` sources the opinionated map's CC numbers.
- [x] **MAP-CONFIG-006**: The system shall require `faders`, `knobs`, `mutes`, and `solos` each to contain exactly 8 CC numbers; a config file with a different count for any of these fields shall fail validation (`PROFILE-LOAD-005`, see `docs/specs/midi-input.md`).
- [x] **MAP-CONFIG-007**: If a Controller Profile's `has_jog_wheel` is true and its `controls:` block omits `jog_wheel`, then that config file shall fail validation (`PROFILE-LOAD-005`, see `docs/specs/midi-input.md`) rather than being loaded with jog-wheel dispatch silently disabled.
- [x] **MAP-CONFIG-008**: The system shall determine Bank N membership (`MAP-BANK-001` through `009`) by positional index — index N-1 in each of a profile's `faders`, `knobs`, and `mutes` lists — not by CC-number arithmetic; a profile whose `knobs`/`mutes` CC numbers bear no arithmetic relationship to its `faders`' shall still bank-derive correctly. Edge audit finding (2026-07-22): the pre-Phase-5 implementation computed bank membership from fixed CC offsets (`+16` for knobs, `+48` for mutes, relative to a fader always at CC N-1), which is incompatible with a config's independently-declared CC lists.

## Opinionated Table

- [x] **MAP-TABLE-001**: For every table entry whose MIDI source is a CC message, the system shall map it to its corresponding Dragonframe OSC message only when the event's MIDI channel equals the active Controller Profile's `default_channel` (15/channel 16 for the Studio, 0/channel 1 for the nanoKONTROL2 — `MIDI-PROFILE-002`/`003`); an otherwise-matching CC control on any other channel shall not match any table entry. This invariant does not apply to the Scene button (`korg_scene` type), when the active profile has one (`has_scene_button`): it matches on type alone — its `channel` field is the controller's own configured Native Mode global-channel ID (see `MIDI-NATIVE-001`), not a stand-in for the profile's `default_channel`, and may legitimately be any value 0–15. Under a profile with no Scene button, no `korg_scene` event is ever produced (`MIDI-EVT-001`, see `docs/specs/midi-input.md`), so this exclusion is moot for that profile.
- [x] **MAP-TABLE-002**: While a fader (CC 0–7) whose target is in OSC encoder mode, or a knob (CC 16–23) whose bank has no axis assigned, changes value, the system shall send its mapped `/dragonframe/encoder/<n>` OSC message (float 0.0–1.0) on every distinct value received, with no debounce. (A fader in OSC axis (direct) mode instead follows `MAP-AXIS-001`/`MAP-AXIS-009`; a knob whose bank has an axis assigned instead follows `MAP-BANK-001`.)
- [x] **MAP-TABLE-003**: When a mapped button-type control (Transport Record, Play, Rewind, Fast Forward, or — under a profile with `has_scene_button` true — the Native Mode Scene button) transitions to pressed, the system shall send that control's mapped one-shot Dragonframe OSC message exactly once per transition. Mute 1–8 follows this same one-shot-per-transition timing, but its message depends on its bank's fader state — see `MAP-BANK-002` (bank has an axis) and `MAP-BANK-004` (bank has no axis, encoder-reset fallback). Stop, Cycle, Solo 1–8, and Previous/Next Marker instead produce a WebSocket command on the same press-edge timing — see `MAP-WS-001` through `MAP-WS-009`, not this spec. Previous Track and Next Track are excluded from this spec as of Phase 6 — they no longer produce Dragonframe OSC output at all; see `MAP-GROUP-003`.
- [x] **MAP-TABLE-005**: If a MIDI event does not correspond to any entry in the active Controller Profile's opinionated table (for the Studio: Record 1–8, Select 1–8, Set Marker, and Return to Zero; for the nanoKONTROL2: Record 1–8 and Set Marker — `MAP-PROFILE-003`), then the system shall produce no OSC output and no log entry for it. This does not apply to Stop, Cycle, Solo 1–8, or Previous/Next Marker — removed from the opinionated table's OSC entries entirely, they instead produce a WebSocket command (`MAP-WS-001` through `MAP-WS-009`), not silence.

## OSC Axis (Direct) Target

- [x] **MAP-AXIS-001**: While a fader targeting an OSC axis (direct) changes value, the system shall send `/dragonframe/axis/{axisname}/gotoPosition,f (position)` on every distinct value, with no debounce, where `position = min + normalized_value * (max - min)` using that mapping entry's configured `min` and `max`.
- [x] **MAP-AXIS-002**: The system shall accept any real-valued `min`/`max` pair for an OSC axis (direct) target — including `min > max` and `min == max` — without validation or rejection.
- [x] **MAP-AXIS-003**: When the user configures an OSC axis (direct) target, the system shall restrict axis name selection to names present in the OSC Listener's discovered list at that moment (`OSC-DISCOVER-005`, `docs/specs/osc-io.md`) and shall not accept an arbitrary or free-text axis name; this restriction applies only at selection time and is not re-enforced afterward against a mapping entry already configured (see `MAP-AXIS-006` for what happens if the axis later disappears). Enforced by the Mapping View UI (picker), not the `MappingEngine` class — see `UI-MAP-004`, `docs/specs/app-ui.md`.
- [x] **MAP-AXIS-006**: If a mapping entry's configured axis name is absent from the OSC Listener's discovered list at the time a `gotoPosition` message would be sent for it, the system shall send the message anyway (targeting that name unconditionally); it shall not withhold the send, warn, or disable the mapping entry based on discovered-list membership.
- [x] **MAP-AXIS-004**: The system shall offer independent OSC axis (direct) target-type selection only for fader controls in this phase; knobs and buttons shall have no independent OSC axis (direct) target-type selection of their own. This does not preclude a knob or Mute button automatically deriving an axis-targeted action from its bank's fader (`MAP-BANK-001`, `MAP-BANK-002`, `MAP-BANK-004`) — derivation is not independent assignment. Solo is unaffected by this spec; it is WebSocket-targeted, not OSC axis (direct)-derived (`MAP-WS-002`).
- [x] **MAP-AXIS-007**: When the engine-wide fader mode (`MAP-AXIS-010`) changes from OSC axis (direct) to OSC encoder, the system shall not remove or otherwise modify any (Bank, Group) axis-target configuration; when it later changes back to OSC axis (direct), every previously-configured (Bank, Group) assignment shall resume driving output exactly as configured, with no re-entry required.
- [x] **MAP-AXIS-008**: The system shall default every fader to OSC axis (direct) mode with no axis name selected, rather than defaulting to OSC encoder mode.
- [x] **MAP-AXIS-009**: While a fader is in OSC axis (direct) mode with no axis name selected, the system shall produce no OSC output for that fader's MIDI events; it shall not fall back to `MAP-TABLE-002`'s OSC encoder channel behavior.
- [x] **MAP-AXIS-010**: The system shall track a single engine-wide setting (`set_fader_mode(axis: bool)`) governing whether every fader is currently in OSC axis (direct) mode or OSC encoder mode, applying identically to all 8 faders — no fader may independently be in a different mode from any other. This is independent of whether an axis name has been selected for any (Bank, Group) pairing; entering OSC axis (direct) mode without any name yet selected for the active Group shall follow `MAP-AXIS-009` (no output), not silently remain in or revert to OSC encoder mode.
- [x] **MAP-AXIS-011**: A call to `set_fader_mode` shall not, by itself, cause a Preset Store write — it does not modify `_group_axis_targets`, so `MAP-STORE-004`'s write trigger does not apply to it.
- [x] **MAP-AXIS-012**: A `set_fader_mode` call that occurs while a fader is mid-gesture (actively producing continuous MIDI values) shall take effect immediately, on the very next MIDI value from that fader; the system shall not defer or ignore it until the gesture ends, mirroring `MAP-GROUP-012`'s mid-gesture Group-switch handling.

## Bank Derivation

Every "Bank N's fader has/has no axis name assigned" condition below means assigned **for the currently active Group** (`MAP-GROUP-002`, see Group Switching below) — a fader's axis assignment is a per-(Bank, Group) value as of Phase 6, not a single fixed value per Bank; these specs describe Bank Derivation's mechanics, which are unchanged, applied to whichever assignment the active Group currently selects.

- [x] **MAP-BANK-001**: While Bank N's fader has a real axis name assigned in the active Group, Knob N shall send `/dragonframe/axis/{axisname}/stepPosition,f (delta)` on every distinct value once a prior reading exists to compare against, with no debounce, where `delta = (raw_value - previous_raw_value) * 0.1` (the change since Knob N's own last reported value, scaled by a fixed factor of `0.1` axis-position units per raw-value increment); the first reading after establishing this state shall produce no send, only a baseline.
- [x] **MAP-BANK-002**: When Bank N's fader has a real axis name assigned in the active Group, Mute N shall send `/dragonframe/axis/{axisname}/setZero` on its press transition (in place of its opinionated encoder-reset target). This recalibrates the axis's zero reference to its current position; it does not move the axis.
- [x] **MAP-BANK-004**: While Bank N's fader has no axis name assigned in the active Group (default state for that (Bank, Group) pairing, or the engine-wide fader mode is OSC encoder — a single setting independent of both Bank and Group, `MAP-AXIS-010`), Knob N and Mute N shall produce their opinionated encoder-channel / encoder-reset targets (`MAP-TABLE-002`/`MAP-TABLE-003`), not the derived actions above. This does not apply to Solo N, which is not bank-derived — see `MAP-WS-002`.
- [x] **MAP-BANK-005**: Knob N's derived `stepPosition` send (`MAP-BANK-001`) shall not be repeated for a repeated identical raw value; this follows directly from `MAP-BANK-001`'s delta formula (a repeat computes to a `0` delta) without a separate dedup mechanism.
- [x] **MAP-BANK-006**: Bank derivation shall not apply to Record N or Select N regardless of Bank N's fader state; they shall remain unmapped (`MAP-TABLE-005`) in every case.
- [x] **MAP-BANK-007**: When the engine-wide fader mode (`MAP-AXIS-010`) transitions between OSC encoder mode and OSC axis (direct) mode in either direction, the system shall discard every Bank's Knob dedup state simultaneously; selecting a different axis name for the active Group on one Bank, while the engine-wide mode remains OSC axis (direct), shall not discard any Bank's dedup state. (A Group switch itself discarding dedup state, a separate trigger from this mode transition, is `MAP-GROUP-011`.)
- [x] **MAP-BANK-008**: While Bank N's fader has a real axis name assigned in the active Group, Knob N's derived `stepPosition` send (`MAP-BANK-001`) shall be clamped such that the tracked axis position it accumulates onto never moves outside the fader's configured `min`/`max` range for that (Bank, Group) assignment (using the lower and higher of the two configured values as the effective bounds, regardless of which was named `min` and which `max`). If the requested delta would carry the tracked position beyond a bound, the system shall send only the reduced delta needed to reach that bound exactly; if the tracked position is already at that bound in the requested direction, the system shall send nothing.
- [x] **MAP-BANK-009**: When computing the current tracked position for the clamping in `MAP-BANK-008`, the system shall prefer Dragonframe's most recently reported live position for that axis name (`OSC-DISCOVER-005`, `docs/specs/osc-io.md`) over its own internally-accumulated estimate; the internal estimate shall be consulted only when no live position reading exists yet for that axis name. Absent both a live reading and a prior internal estimate, the tracked position shall be assumed to start at the fader's configured lower bound.

## Group Switching

Introduced in Phase 6 (`docs/llds/static-mapping.md § Group Switching`). "Bank" keeps its pre-existing meaning throughout this document (a fixed 1-of-8 channel-strip grouping); "Group" is the new, orthogonal, switchable dimension these specs define.

- [x] **MAP-GROUP-001**: The system shall support exactly 5 Groups, numbered 1–5; no configuration or discovered state shall change this count.
- [x] **MAP-GROUP-002**: The system shall track one active-Group value, `_active_group`, ranging 1–5 inclusive, initialized to `1` on `MappingEngine` construction.
- [x] **MAP-GROUP-003**: When Previous Track or Next Track (`group_keys.previous`/`.next`, sourced from the active profile's `controls.transport` `previous_track`/`next_track` CCs) transitions to pressed, the system shall step `_active_group` by one in the corresponding direction (per `MAP-GROUP-004`'s boundary rule) exactly once per transition, debounced identically to `MAP-DEBOUNCE-001`, and shall produce no OSC message and no WebSocket command for that transition.
- [x] **MAP-GROUP-004**: When stepping `_active_group` past either boundary (`MAP-GROUP-003`), the system shall wrap — Next from Group 5 shall move to Group 1; Previous from Group 1 shall move to Group 5 — rather than clamping at the boundary.
- [x] **MAP-GROUP-005**: The system's `process()` shall evaluate the Previous Track/Next Track special-case (`MAP-GROUP-003`) immediately after the channel-match check and before the jog-wheel special-case (`MAP-JOG-000`ff.) and every other dispatch path, such that a CC collision between a Group-switch key and any other control's CC resolves in favor of Group switching.
- [x] **MAP-GROUP-006**: Where a Controller Profile's `controls.transport` declares only one of `previous_track`/`next_track`, the system shall support Group stepping only in the declared direction; the undeclared direction's row/behavior shall not exist, matching `MAP-CONFIG-004`'s "absent, not disabled" treatment for any other omitted `transport` key. This combination shall not fail config validation.
- [x] **MAP-GROUP-007**: The system shall not reset `_active_group` on an ordinary MIDI reconnect clear (`MIDI-EVT-004`, see `docs/specs/midi-input.md`); it shall reset `_active_group` to `1` only on a Controller Profile switch (`MAP-PROFILE-004`) and on `MappingEngine` construction (`MAP-GROUP-002`).
- [x] **MAP-GROUP-008**: The system shall store each fader's OSC axis (direct) target keyed by `(fader_key, group)`, not by `fader_key` alone; `set_axis_target` and `axis_target` shall each take an explicit `group` parameter (1–5) rather than implicitly operating on `_active_group`, such that a caller may read or write any Group's assignment regardless of which Group is currently active.
- [x] **MAP-GROUP-009**: `clear_group_axis_target(key, group)` shall clear only the named Group's assignment for that fader, leaving the engine-wide fader mode (`MAP-AXIS-010`) and every other Group's assignment for that fader (and every other fader entirely) untouched. It shall not be the mechanism for changing the engine-wide fader mode — that is `set_fader_mode` alone (`MAP-AXIS-010`/`011`).
- [x] **MAP-GROUP-010**: `set_axis_target(key, group, ...)` and `clear_group_axis_target(key, group)` shall discard `key`'s (and its paired Knob's) dedup/position state (per `MAP-BANK-007`'s discard mechanism) only when `group == self._active_group`; a write to a non-active Group's assignment shall leave the active Group's in-flight dedup/position state untouched.
- [x] **MAP-GROUP-011**: When `_active_group` changes (`MAP-GROUP-003`), the system shall discard the dedup/position state (`MAP-BANK-007`'s mechanism) for every Bank currently in OSC axis (direct) mode (`MAP-AXIS-010`), and shall not discard state for any Bank currently in OSC encoder mode.
- [x] **MAP-GROUP-012**: The system shall not defer, ignore, or otherwise special-case a Previous Track/Next Track press that arrives while a fader is mid-gesture (actively producing continuous MIDI values); the next MIDI value received after the Group switch shall be evaluated against the newly-active Group's assignment immediately.

## Preset Store

Introduced in Phase 6 (`docs/llds/static-mapping.md § Preset Store`) — the first implementation of the general Preset Store concept named in `docs/high-level-design.md`'s System Design, scoped narrowly to the (Bank, Group) axis-assignment table.

- [x] **MAP-STORE-001**: The system shall persist each Controller Profile's (Bank, Group) axis-assignment table to its own file under `~/Documents/DragonMIDI/configurations/`, named after the profile, keyed by profile name plus positional Bank index (1–8, per `MAP-CONFIG-008`, resolved to a fader key via the new ordered `ControllerProfile.bank_fader_keys` field) plus Group index (1–5), not by raw CC number. File I/O and validation shall live in a dedicated `preset_store.py` module decoupled from `MappingEngine`'s internal types, matching the existing separation between `controller_profile_loader.py` and `controller_profile.py`.
- [x] **MAP-STORE-002**: When a Controller Profile becomes active (at app launch for the default/last-selected profile, or on a Controller Profile switch), the system shall load that profile's persisted (Bank, Group) axis-assignment table, if a file exists for it, via `MappingEngine.load_group_axis_targets(bank_axes)` — a bulk-populate method that writes directly into `_group_axis_targets` without triggering `MAP-STORE-004`'s save-on-change behavior; absence of a file shall not be treated as an error. The reverse translation, for saving, shall be available via `MappingEngine.dump_group_axis_targets()`.
- [x] **MAP-STORE-003**: When loading a persisted configuration file, the system shall validate each entry's Bank index (1–8), Group index (1–5), and value shape (axis name string, numeric min/max) before applying it; an entry failing validation shall be skipped, with a logged warning, and shall not prevent the file's other valid entries from loading.
- [x] **MAP-STORE-004**: The system shall write the active Controller Profile's complete (Bank, Group) axis-assignment table to its configuration file on every `set_axis_target` or `clear_group_axis_target` call, with no separate save step, rewriting the whole table rather than diffing it. `set_fader_mode` shall not trigger this write (`MAP-AXIS-011`).
- [x] **MAP-STORE-005**: If a configuration file write fails (e.g. disk full, permissions), the system shall log the failure and shall not retry it, surface a new status indicator, or block continued use of the in-memory (Bank, Group) axis-assignment table for the remainder of the session.
- [x] **MAP-STORE-006**: The system shall never persist `_active_group`; every load (`MAP-STORE-002`) shall leave `_active_group` at whatever `MAP-GROUP-002`/`MAP-GROUP-007` already established (`1` on launch or profile switch), regardless of the configuration file's contents.
- [x] **MAP-STORE-007**: `preset_store.py`'s `load_group_axis_targets`/`save_group_axis_targets` shall be plain synchronous functions, with no async or background-thread I/O; the coordinating call site (`app.py`, on a Controller Profile switch) shall invoke `MappingEngine.set_profile()` (`MAP-PROFILE-004`) and the subsequent `load_group_axis_targets`/`MappingEngine.load_group_axis_targets` pair in the same synchronous handler, such that no caller can observe `_group_axis_targets` in a cleared-but-not-yet-reloaded state between the two calls.

## Jog Wheel Frame Stepping

- [x] **MAP-JOG-000**: This entire section applies only while the active Controller Profile's `has_jog_wheel` is true (the nanoKONTROL Studio). Under a profile with `has_jog_wheel` false (the nanoKONTROL2), `process()` shall skip the jog-wheel special-case dispatch entirely — gated on the feature flag, not on the absence of CC110 from that profile's opinionated map.
- [x] **MAP-JOG-001**: While the jog wheel (CC 110, on the active profile's `default_channel`) reports a sign-magnitude relative value in the clockwise range (`1`–`63`), the system shall send `/dragonframe/stepForward` exactly once per received message, regardless of the value's magnitude.
- [x] **MAP-JOG-002**: While the jog wheel (CC 110, on the active profile's `default_channel`) reports a sign-magnitude relative value in the counterclockwise range (`65`–`127`), the system shall send `/dragonframe/stepBackward` exactly once per received message, regardless of the value's magnitude.
- [x] **MAP-JOG-003**: When the jog wheel reports a sign-magnitude relative value of `0` or `64`, the system shall produce no OSC output.
- [x] **MAP-JOG-004**: The system shall not apply `MAP-DEBOUNCE-001`'s debounce window, nor any dedup-on-repeated-value logic, to jog wheel messages; each message qualifying under `MAP-JOG-001` or `MAP-JOG-002` shall produce its OSC send independent of any prior message's raw value, including a repeated identical value.
- [x] **MAP-JOG-005**: The system shall allocate no per-control state (previous-value, pressed-state, or last-fired) for the jog wheel entry.

## Jog Wheel Keystroke Output (Arc Motion Control)

- [x] **MAP-JOGKEY-000**: This entire section applies only while the active Controller Profile's `has_jog_wheel` is true, matching `MAP-JOG-000` — the Arc Motion Control keystroke output exists solely as the jog wheel's second output.
- [x] **MAP-JOGKEY-001**: While the jog wheel (CC 110, on the active profile's `default_channel`) reports a sign-magnitude relative value in the clockwise range (`1`–`63`), the system's `process_keystroke()` shall return `KeyCombo(frozenset({"alt", "shift"}), "right")`.
- [x] **MAP-JOGKEY-002**: While the jog wheel (CC 110, on the active profile's `default_channel`) reports a sign-magnitude relative value in the counterclockwise range (`65`–`127`), the system's `process_keystroke()` shall return `KeyCombo(frozenset({"alt", "shift"}), "left")`.
- [x] **MAP-JOGKEY-003**: When the jog wheel reports a sign-magnitude relative value of `0` or `64`, or the event does not match the jog wheel, `process_keystroke()` shall return `None`.
- [x] **MAP-JOGKEY-004**: The system shall evaluate `process_keystroke()` for a jog wheel event independent of `process()`'s OSC output for that same event — both shall be evaluated, with neither suppressing the other based on which one would have an effect in Dragonframe's current workspace.
- [x] **MAP-JOGKEY-005**: `process_keystroke()` shall not apply debounce or dedup logic to jog wheel messages, matching `MAP-JOG-004`.
- [x] **MAP-JOGKEY-006**: `process_keystroke()` shall respect the same channel match invariant as `process()` (`MAP-TABLE-001`) for the jog wheel's CC-sourced event, against the active profile's `default_channel`.
- [x] **MAP-JOGKEY-007**: `process_keystroke()` shall allocate no per-control tracked state; calls to it shall not affect `tracked_controls()` or any state consulted or mutated by `process()`.

## WebSocket-Targeted Controls

All four controls below exist on both Controller Profiles (unlike the jog wheel and Scene button); only the `_Key` match tuples' channel component varies, per the active profile's `default_channel` (`MAP-TABLE-001`) — none of these specs are feature-gated.

- [x] **MAP-WS-001**: When Stop (CC 42, on the active profile's `default_channel`) transitions to pressed, the system's `process_websocket()` shall return `WebSocketCommand("E-Stop")` exactly once per transition, debounced identically to `MAP-DEBOUNCE-001`. Stop shall not appear in `OPINIONATED_MAP`; `process()` shall return `None` for it.
- [x] **MAP-WS-002**: When Solo N (CC 32–39, on the active profile's `default_channel`, N = 1–8) transitions to pressed, the system's `process_websocket()` shall return `WebSocketCommand(f"select-AX{N + 8 * (g - 1)}")`, where `g` is the currently active Group (`MAP-GROUP-002`), exactly once per transition, unconditional on Bank N's fader/axis state. (Prior to Phase 6, with no Group dimension, this was `select-AX{N}` — equivalent to `g = 1`.)
- [x] **MAP-WS-003**: When Cycle (CC 46, on the active profile's `default_channel`) transitions to pressed while `axis_count` (`len(axis_positions or {})`) is greater than `0`, the system's `process_websocket()` shall increment `_cycle_index = (_cycle_index + 1) % axis_count` and return `WebSocketCommand(f"select-AX{_cycle_index + 1}")` exactly once per transition.
- [x] **MAP-WS-004**: If Cycle transitions to pressed while `axis_count` is `0`, then `process_websocket()` shall return `None` and shall not advance `_cycle_index`.
- [x] **MAP-WS-005**: The system shall initialize `_cycle_index` to `-1` on `MappingEngine` construction and reset it to `-1` on every MIDI reconnect clear (`MIDI-EVT-004`, `docs/specs/midi-input.md`) or Controller Profile switch (`MAP-PROFILE-004`), matching `MAP-STATE-002`'s reconnect hygiene for other tracked state.
- [x] **MAP-WS-006**: When Previous Marker (CC 61, on the active profile's `default_channel`) transitions to pressed, the system's `process_websocket()` shall return `WebSocketCommand("Jog All", operation="+", params=(-1,))` exactly once per transition.
- [x] **MAP-WS-007**: When Next Marker (CC 62, on the active profile's `default_channel`) transitions to pressed, the system's `process_websocket()` shall return `WebSocketCommand("Jog All", operation="+", params=(1,))` exactly once per transition.
- [x] **MAP-WS-008**: Stop, Solo 1–8, Cycle, and Previous/Next Marker shall share the same press-edge/80ms-debounce state (`_pressed_state`/`_last_fired`) that `process()` uses for other button-type controls; `process_websocket()` shall not maintain separate debounce state for these keys.
- [x] **MAP-WS-009**: Stop, Cycle, and Previous/Next Marker shall be removed from `OPINIONATED_MAP` entirely; `process()` shall return `None` for these keys.

## Debounce and State

- [x] **MAP-DEBOUNCE-001**: If a second press-edge for the same button-type control arrives within 80ms of its previous press-edge, then the system shall drop the second press-edge rather than queueing it or re-firing it after the window closes.
- [x] **MAP-STATE-001**: The system shall allocate previous-value state only for controls present in the opinionated table; controls not in the table shall never have state allocated for them.
- [x] **MAP-STATE-002**: While a mapped button-type control has produced no MIDI message since app start (or since the last reconnect clear per `MIDI-EVT-004`, `docs/specs/midi-input.md`, or Controller Profile switch per `MAP-PROFILE-004`), the system shall treat it as not-pressed, such that a control already held down at that time requires one release-then-press cycle before its first press-edge fires.
